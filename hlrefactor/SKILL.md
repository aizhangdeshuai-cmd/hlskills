# 旧项目 AI 重构流程

> 属于 `hlskills` 技能系统。核心原则：**先建测试护城河，再逐模块替换，每次回归验证，不破坏现有功能。**

## 通用纪律
> **路径规范**：本文件涉及的 `docs/` 路径命名遵循 `hlpm-product/path-conventions.md` 中央规范（命名/深度/特例）。


- **铁律：没有测试就别重构** — 重构前必须先为待改模块补充测试
- **每次只改一个模块** — 改完立即跑全量回归，通过后再改下一个
- **受阻立即停止** — 3 次修复尝试失败 = 质疑重构方案
- **证据先于断言** — 任何步骤完成后验证

## 重构流程

### 第一阶段：理解旧项目（3 步）
0. **加载已有分析文档** — 如项目之前已被 AI 分析过（`/hllegacy`），先加载已有分析文档：技术栈清单 / 模块架构说明 / 逻辑架构图 / API清单 / 前端设计规范 / 质量基线 / 技术债清单，避免重复扫描
1. **代码全景分析** — `architect` 在已有文档基础上增量分析，输出补充发现：
   - 模块清单：每个模块的职责、文件列表、代码规模
   - 依赖关系：模块间调用关系、外部依赖（库/服务/DB）
   - 数据流：请求→处理→存储→响应的完整链路
   - 技术债清单：过时库、反模式、安全漏洞、无测试模块、硬编码配置
2. **架构现状文档** — `architect` 基于分析结果，生成：
   - ER 图（数据库表关系）
   - 逻辑架构图（模块划分与调用关系）
   - API 端点清单（路径、方法、调用方、返回类型）
3. **风险标注** — 对每个模块标注风险等级：
   - 🔴 高危：核心路径 + 紧耦合 + 无测试覆盖
   - 🟡 中危：非核心路径或单侧（有测试或有隔离）
   - 🟢 低危：独立模块，边界清晰，测试覆盖好

### 第二阶段：制定重构策略（4 步）
4. **明确重构目标** — 确定要解决什么：
   - 性能优化 / 可维护性提升 / 安全性增强 / 框架版本升级 / 技术栈迁移
5. **选择重构策略**：
   | 策略 | 适用场景 | 特点 |
   |------|---------|------|
   | 绞杀者模式（Strangler Fig） | 大型系统逐步改造 | 新功能新模块写，旧模块逐步替换 |
   | 增量模块替换 | 中型系统，模块边界清晰 | 逐模块用新代码替换，每次只改一个 |
   | 大爆炸（Big Bang） | 🚫 **不推荐** | 一次性全改，风险极高 |

6. **模块优先级排序** — 按以下顺序排列：
   - 先：无依赖的叶子模块（基础设施层）
   - 中：中度耦合的中间层模块
   - 后：核心编排层（依赖其他模块最多）
7. **ADR 记录** — 对每个关键技术决策以 ADR 格式记录到 `docs/adr/`

### 第三阶段：建立测试护城河（2 步）
8. **补充集成测试** — `test-engineer` 为待重构模块编写集成测试：
   - 覆盖模块的公开接口（输入→输出验证）
   - 覆盖边界条件和异常路径
   - 目标是让重构后能立即验证"行为不变"
9. **建立测试基线** — `qa-tester` 运行全量测试，记录：
   - 测试总数、通过数、失败数
   - 覆盖率报告（分支/函数/行）
   - **基线快照**：重构前全部通过才算就绪

### 第四阶段：逐模块执行重构（5 步）
10. **Git 工作区准备** — 为当前模块创建隔离工作区
11. **逐模块重构** — `executor` 执行，每次只改一个模块：
    - 保持公开接口不变（除非有充分理由）
    - 遵循编码标准、API 设计规范、错误处理规范
    - 如有 DB 变更，遵循数据库迁移规范
12. **全量回归测试** — `qa-tester` 每次模块变更后跑全量测试：
    - 与基线对比，不得新增失败
    - 如有失败，立即修复，不进入下一个模块
13. **代码审查** — `code-reviewer` 审查每次变更：
    - 审查点：行为是否不变、是否有新依赖、是否有安全风险
14. **合并与快照** — 通过后合并到主分支，为下一模块创建新快照

### 第五阶段：收尾与交付（7 步）
15. **安全审查** — `security-reviewer` 全量安全审查（重构可能改变安全边界）
16. **性能对比** — 对比重构前后关键指标（响应时间、内存、构建时间）
17. **文档更新** — 更新技术设计文档、接口文档、数据库设计、ADR
18. **变更日志** — 输出重构前后模块对比：改了什么、为什么改、影响范围
19. **迁移指南** — 如有破坏性变更（API/DB/配置），提供开发者迁移步骤
20. **生产审计** — `verifier` 执行生产审计，评分 >= 70 方可上线
21. **发布部署** — 按发布流程和部署规范执行上线

## 重构安全规则

| 规则 | 说明 |
|------|------|
| **先测试后重构** | 没有测试覆盖的模块不得开始重构 |
| **一次一个模块** | 绝不并行重构多个模块 |
| **每次回归** | 每次变更后跑全量测试，不得跳过 |
| **接口不变** | 尽量保持公开接口不变，变更须记录 |
| **可回滚** | 每个模块重构后可独立回滚 |
| **文档同步** | 代码改了什么文档就更新什么 |

## 交付文档清单

重构交付时必须包含：

### 分析文档
1. **代码分析报告** — 模块清单、依赖关系、数据流、技术债清单、风险标注
2. **架构现状文档** — ER 图、逻辑架构图、API 清单

### 策略文档
3. **重构策略与计划** — 目标、策略选择、模块优先级、时间线
4. **架构决策记录** — 重构过程中的关键技术选型

### 技术文档
5. **重构后技术设计文档** — 更新后的接口文档、数据库设计、逻辑架构图
6. **测试基线报告** — 重构前基线 vs 重构后对比

### 交付文档
7. **变更日志** — 重构前后模块对比（改了什么、为什么改、影响范围）
8. **迁移指南** — 破坏性变更的迁移步骤
9. **性能对比报告** — 重构前后关键指标对比
