# 新项目分析流程

> 属于 `hlskills` 技能系统。在新项目启动前，先回答"要不要做、值不值得做"。分析完成后衔接 `/hlpm` 进入开发。

## 通用纪律

- 分析结论必须有数据支撑，不得凭直觉判断
- 竞品分析至少覆盖 3 个直接竞品
- 每个高风险必须有应对预案
- 最终 Go/No-Go 决策由用户做出，不接受 Agent 自行裁决

## 角色与 Agent 映射

| 角色 | Agent | 文件 |
|------|-------|------|

| 技术评估 | `architect` | `agents/architect.md` |

## 分析流程

### 第一阶段：市场分析（4 步）
1. **市场调研** — `analyst` 调研目标市场：
   - 市场规模（TAM / SAM / SOM）、增长趋势、行业阶段
   - 用户核心痛点与未被满足的需求
   - 参考 `market-research` 技能获取市场数据
   - 📸 **如用户提供参考文件（含图片）**：优先用模型原生多模态识别；不支持则用 Read 工具读取图片（支持 PNG/JPG/WebP/PDF），识别参考产品截图/市场图表/竞品界面
2. **竞品深度分析** — `analyst` 至少分析 3-5 个竞品：
   - 功能矩阵对比（核心功能有无、优劣）
   - 定价策略对比（免费/订阅/按量/企业版）
   - 市场份额与用户口碑
   - 每个竞品的核心优势与弱点
3. **目标用户画像** — `analyst` 输出用户画像：
   - 用户群体（C端/B端/内部）、使用场景、决策链
   - 核心诉求排序（用户最在意的 3 件事）
4. **差异化定位** — `analyst` 明确：
   - 我们的独特优势（功能/价格/体验/生态/数据）
   - 为什么用户选我们而不是竞品
   - 如果不能一句话说清楚差异化，标注为 🔴 风险

### 第二阶段：可行性分析（4 步）
5. **技术可行性** — `architect` 评估：
   - 推荐技术栈（语言/框架/数据库/基础设施）
   - 关键难点识别（有现成方案 / 需要自研 / 当前不可行）
   - 第三方依赖与集成复杂度
6. **资源可行性** — `planner` 估算：
   - 人力需求（前端/后端/设计/测试各需几人）
   - 技能要求（团队是否具备，缺少什么）
   - 时间线估算（乐观/最可能/悲观，含关键里程碑）
7. **合规可行性** — `analyst` 检查：
   - 行业法规要求（如金融/医疗/教育）
   - 数据隐私合规（GDPR / 个人信息保护法）
   - 行业准入或许可要求
8. **商业可行性** — `analyst` 分析：
   - 预估开发成本（人月 × 人天成本 + 基础设施 + 第三方服务）
   - 收入模型（订阅/交易抽成/广告/授权/免费增值）
   - 简单盈亏：预计多久收回成本

### 第三阶段：风险评估（3 步）
9. **风险识别** — `analyst` + `architect` 联合识别：
   - 技术风险（新技术不成熟、性能瓶颈、依赖不稳定）
   - 市场风险（需求假设错误、竞品碾压、用户获取成本高）
   - 资源风险（关键人离职、招聘困难、资金断裂）
   - 合规风险（法规变化、准入受阻）
   - 依赖风险（第三方服务涨价/关闭/变更）
10. **风险分级** — 对每个风险标定：
    - 严重性：🔴 严重（项目失败） / 🟡 中等（延期或超支） / 🟢 轻微（可接受）
    - 概率：高（>50%） / 中（20-50%） / 低（<20%）
    - 影响：对进度/成本/质量的具体影响
11. **应对预案** — 对每个 🔴 高风险制定：
    - 预防措施（如何降低概率）
    - 应急预案（发生后怎么办）
    - 触发条件（什么信号出现时激活预案）

### 第四阶段：决策（3 步）
12. **方案对比** — `analyst` 呈现可选方案：
    | 方案 | 周期 | 成本 | 风险 | 灵活性 |
    |------|------|------|------|--------|
    | 自研 | 长 | 高 | 可控 | 最高 |
    | 买 SaaS | 短 | 中 | 供应商依赖 | 低 |
    | 开源二次开发 | 中 | 中 | 社区依赖 | 中 |
    | 外包 | 中 | 高 | 质量控制难 | 低 |
13. **投入产出分析** — `planner` 汇总：
    - 总投入（人/时间/钱/机会成本）
    - 预期产出（收入/效率提升/战略价值）
    - 回本周期与 ROI 估算
14. **Go/No-Go 决策** — 提交给用户：
    - 综合评估结论（推荐/有条件推荐/不推荐）
    - 关键假设与验证方法
    - 🚨 用户决策：Go（进入启动准备）/ No-Go（终止）/ Conditional（先验证假设再决定）

### 第五阶段：启动准备（4 步）
15. **项目章程** — `planner` 起草：
    - 项目名称、目标、范围边界（做什么 + 不做什么）
    - 关键干系人、成功标准（可量化指标）
    - 约束条件（预算/时间/技术栈限制）
16. **初始需求文档** — `analyst` 生成内容 → 写入文件：
    - 核心功能范围（MVP 必须有什么）
    - 非功能需求（性能/安全/可用性/可维护性）
    - 暂缓功能（后续版本再做的）
17. **技术选型建议** — `architect` 输出：
    - 推荐技术栈及理由（语言/框架/数据库/部署方式）
    - 关键技术决策以 ADR 格式记录到 `docs/adr/NNNN-<slug>.md`（4 位编号，详见 `hladr`）
18. **衔接开发流程** — 将分析结论传递到 `/hlpm`：
    - 市场分析报告 → hlpm 竞品分析
    - 初始需求 + 项目章程 → hlpm 需求分析 + PRD
    - 技术选型 → hlpm 技术设计文档

## 交付文档清单

新项目分析交付时必须包含：

### 市场与竞品
1. **市场分析报告** — 市场规模、趋势、用户痛点、目标画像
2. **竞品分析矩阵** — 功能对比、定价对比、优劣势、差异化定位

### 可行性与风险
3. **技术可行性报告** — 技术栈建议、关键难点、现成方案对比
4. **资源与排期估算** — 人力需求、技能要求、时间线
5. **风险评估登记册** — 风险清单、分级、概率、影响、应对预案
6. **投入产出分析** — 成本估算、收益预测、ROI

### 决策与启动
7. **Go/No-Go 决策记录** — 结论、理由、验证假设
8. **项目章程** — 目标、范围、干系人、成功标准、约束
9. **初始需求文档** — 核心功能、非功能需求、暂缓功能
10. **技术选型建议** — 推荐技术栈 + ADR
