# 评审与审查机制

## 通用纪律：完成前验证

审查结论必须有证据支持，禁止"看起来没问题"等模糊判断。证据先于断言。

---

## 审查阶段

每个评审/审查阶段调用对应 Agent，输出结论：

| 阶段 | Agent | 输出要求 | 适用流程 |
|------|-------|---------|---------|
| PRD评审 | `analyst` + `architect` | 通过 / 修改建议 / 驳回 | 新需求 |
| 设计评审 | `designer` + `executor` | 通过 / 修改建议 / 驳回（含无障碍检查） | 新需求 |
| 用例评审 | `test-engineer` + `analyst` | 通过 / 修改建议 / 驳回 | 新需求 |
| 修复方案评审 | `architect` + `executor` | 通过 / 修改建议 / 驳回 | Bug修复 |
| 代码审查 | `code-reviewer` | 严重性分级：严重/重要/次要 + 是否可合并 | 两者 |
| 安全审查 | `security-reviewer` | 漏洞清单 + 风险等级 + 修复建议 | 两者 |
| 生产审计 | `verifier` | 风险评分(0-100) + 风险视角 + 证据清单 | 两者 |

## 代码审查详细规范

### 审查标准
参考 `requesting-code-review` 技能：
- 严重性分级：**严重**（必须修复）→ **重要**（应该修复）→ **次要**（可有可无）
- 输出格式：优势 → 问题（按严重性，带 file:line）→ 建议 → 评估
- 评估结论：准备合并？是 / 否 / 需要修复

### 行为规范
参考 `receiving-code-review` 技能：
- **禁止**表演性同意（"您完全正确！"、"好观点！"、"感谢您..."）
- 用技术推理反驳，或用修复展示反馈响应
- 对不清楚的反馈，先确认所有项目再实施

### 审查检查项
- 实现是否符合技术设计文档
- 是否遵循编码标准（命名、不可变性、类型安全、文件组织）
- 代码味道：魔法数字、复杂三元表达式、重复代码、未处理错误路径
- 是否通过所有测试
- 安全性：无密钥泄露、无注入漏洞、认证鉴权正确

### 结构性问题检查（融合 gstack /review）

gstack /review 识别的结构性问题类型，审查时必须覆盖：

- **SQL 安全**：拼接 SQL 字符串、缺少参数化查询、N+1 查询问题
- **LLM 信任边界**：用户输入直通 LLM prompt、注入可执行指令的文本
- **条件副作用**：if/else 分支中隐藏的 API 调用、状态变更、副作用
- **竞态条件**：异步操作间缺少锁定或顺序保证
- **异常路径未处理**：try/catch 为空、错误吞没

## 安全审查详细规范

参考 `security-review` 技能：
- 安全漏洞扫描（OWASP Top 10）
- 密钥与敏感信息检查
- 认证与鉴权链路审查
- 输出：漏洞清单 + 风险等级 + 修复建议

### 补充检查项（融合 gstack /cso）

gstack /cso 首席安全官审查可独立运行，或作为 hlreview 安全审查的补充：

- **秘密考古**：扫描 git 历史中的 API Key、Token、密码提交，检查 `.env` 是否被误提交
- **依赖供应链**：审查依赖项是否有已知 CVE，是否使用固定版本（非 `latest`）
- **CI/CD 管道安全**：检查 CI 配置中的密钥注入方式、分支保护规则、部署权限
- **LLM/AI 安全**：Prompt 注入防护、输出内容过滤、模型访问控制

运行方式：在终端执行 `Skill gstack cso` 加载 gstack /cso 技能进行独立安全审查。

## 跨模型第二意见（融合 gstack /codex）

对于高风险的代码变更，可通过 gstack /codex 获取独立模型的第二意见：

- **代码评审模式**：提交 diff 给 codex 进行独立分析，发现 Claude 可能遗漏的问题
- **挑战模式（Adversarial）**：Codex 尝试找出代码中的安全漏洞和逻辑错误
- **咨询模式**：针对特定问题点追问 codex，获取不同视角的建议

使用时机：安全关键代码（认证、支付、权限）、大型重构、生产环境热修复。
运行方式：在终端执行 `Skill gstack codex` 加载 gstack /codex 技能。

## 生产审计详细规范

参考 `production-audit` 技能：
- **风险评分**：0-49 受阻 / 50-69 有风险 / 70-84 可发布 / 85-100 强
- **风险视角**：安全与认证、数据完整性、支付与Webhook、运营、用户体验
- 输出：证据清单 + 缺失证据 + 阻碍项/高风险修复建议

## 接收审查反馈规范

审查完成后，被审查方必须遵循以下流程处理反馈：

### 响应流程
1. **阅读** — 逐条阅读所有反馈，理解每一条的含义
2. **验证** — 在实施前验证反馈是否技术上正确
3. **评估** — 判断每条反馈的合理性：接受 / 有理有据反驳 / 需澄清
4. **回应** — 对不清楚的反馈提出具体技术问题，不完全澄清前停止实施
5. **实施** — 按优先级顺序修复：阻碍性问题 → 简单修复 → 复杂修复

### 禁止事项
- **禁止表演性同意**："你说得完全正确！"、"好观点！"、"谢谢你的反馈！"
- **禁止盲目接受**：每个反馈先验证再行动
- **禁止选择性忽略**：严重问题不得以"之后处理"为借口跳过

### 何时提出异议
- 反馈违反 YAGNI 原则（过度设计）
- 反馈引入实际不存在的场景
- 反馈技术上不正确
- 反馈与用户的架构决策冲突

### 多反馈处理顺序
1. 先处理阻碍性问题（CRITICAL/HIGH置信度+高严重性）
2. 再处理简单修复（单文件、低风险）
3. 最后处理复杂修复（跨文件、需架构变更）
4. 每修复一项，验证其不影响其他反馈

### 修复后二次验证
反馈修复完成后，必须执行：
- **推送修复到原审查方**：将修复代码提交到审查方进行二次审查
- **运行完整测试套件**：确保修复未引入回归
- **更新审查结论**：原审查方更新审查状态为"通过"或"仍需修改"
