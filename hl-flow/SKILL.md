# 新需求开发流程

> ## ⚠️ 流程纪律的执行机制
>
> 本流程的"硬性关卡""返回上一步""强同步规则"是 markdown 文本纪律,Claude Code 不会机械地强制执行。Agent 加载后会**自觉遵守**,但如 Agent 模型弱或 prompt 冲突,关卡可能失效。
>
> **真实约束**:文档本身 + Agent 自觉 + 用户手动打断。
> **不是**:Claude Code runtime hook 拦截 + 状态机门禁。
>
> 如发现 Agent 在第 X 步没停下来等你确认 / 直接跳过第 Y 步 / 没返回上一步重做,请**手动打断**让它重读 SKILL.md 对应章节。
>
> 详细能力边界见仓库 `README.md` 「⚠️ 能力边界声明」段。


## 通用纪律：完成前验证
> **路径规范**：本文件涉及的 `docs/` 路径命名遵循 `hlpm/path-conventions.md` 中央规范（命名/深度/特例）。


**任何阶段标记"完成"前，必须执行以下验证规则：**

- 识别能证明该阶段完成的验证命令，完整运行该命令
- 读取完整输出并检查退出码，确认通过后方可声明完成
- **绝对禁止**："应该通过了"、"看起来正确"、"之前检查过了"、"应该没问题"、"可能没问题"
- **铁律**：证据先于断言，无一例外。未经新鲜验证证据，不得声称完成

### 受阻停止纪律
- 任何步骤遇到阻塞，**立即停止**，不得猜测或强行绕过
- 向产品经理（`analyst`）报告阻塞情况
- 产品经理无法解决时，立即向用户提问
- 3 次修复尝试失败 = 质疑架构方案，不尝试第 4 次变体修复

## 角色与 Agent 映射

| 角色 | Agent | 用途 |
|------|-------|------|
| 产品经理 | `analyst` / `planner` | 需求分析、任务规划与分配、流程统筹 |
| UI/UX设计师 | `designer` | 界面设计、交互设计、设计评审 |
| 前端开发 | `executor` | 前端页面与组件开发 |
| 后端开发 | `architect` / `executor` | 后端架构设计与API开发 |
| 代码审查 | `code-reviewer` | 代码质量审查 |
| 安全审查 | `security-reviewer` | 安全漏洞扫描 |
| 测试 | `test-engineer` / `qa-tester` | 测试用例编写与执行 |
| 验证 | `verifier` | 交付前最终验证 |
| 用户（我） | — | 关键节点确认者，未确认不得进入下一阶段 |

## 新需求开发流程

### 第零阶段：加载项目上下文
0. **加载已有文档** — 在分析新需求之前，**先读取项目已有知识**，避免重复分析：
   - 如存在 `.hl/memory/` 目录 → Read 全部 memory 文件（project.md / techstack.md / architecture.md / conventions.md / working.md）
   - 如存在已有分析文档（竞品分析报告 / PRD / 技术设计文档 / ADR）→ Read 加载
   - 📎 迭代需求场景：**必须先理解项目现状，再分析新需求**。跳过此步会导致脱离上下文

### 第一阶段：需求阶段
1. **需求分析** — `analyst` 基于已有文档上下文，理解并梳理新需求，如用户提供需求文档（含图片）：
   - 使用 Read 工具读取文本内容
   - 📸 **分析文件中的图片**：优先用模型原生多模态能力识别图片中的界面草图/流程图/参考截图/架构图；不支持则用 Read 工具读取（支持 PNG/JPG/WebP/PDF）
   - 将图片内容与文字需求合并分析，形成完整需求理解
2. **竞品分析**（新项目启动时必须执行） — `analyst` 进行市场竞品调研，参考可选外部 `market-research` 技能（未安装时 `analyst` 直接基于联网搜索与公开资料自行调研）：
   - 识别直接竞品与间接竞品
   - 分析竞品核心功能、差异化优势、定价策略
   - 输出竞品分析报告，明确自身产品定位与差异化方向
   - 📎 **如从 `/hlpm` 进入**：导入 hlpm 的竞品分析报告和初始需求文档，跳过本步及第3步，直接进入 PRD 评审
3. **PRD编写** — 产品经理（`analyst`）生成 PRD 内容 → 写入文件（如从 hlpm 导入已有需求文档，跳过编写直接评审）
4. **PRD评审** — 联合会议评审，`analyst` 主持，全体参与：
   - `analyst`（产品）— 确认需求完整性与优先级
   - `architect`（后端）— 技术可行性与架构影响
   - `executor`（前端）— UI 可实现性与交互合理性
   - `designer`（UI/UX）— 设计可行性与用户体验
   - `test-engineer`（测试）— 可测试性与边界条件覆盖
   - 各角色各自输出评审意见，产品经理汇总。无法达成一致的提交用户决策
5. **UI/UX设计** — `designer` 进行界面和交互设计，设计决策流程：

   **第一步：检查现有设计规范**
   - 项目已有前端设计规范文档（设计系统/组件库/色彩/字体/间距）？
     - ✅ 有 → 严格遵循，不得偏离既定规范
     - ❌ 无 → 进入第二步

   **第二步：是否先整理设计规范**
   - 询问用户：是否先整理/建立本项目的设计规范？
     - ✅ 整理 → 调用 `hllegacy` 第11步：读取当前系统前端源码（CSS变量/Tailwind配置/组件源码），提取字体/色彩/按钮/表格/表单/弹框/筛选/图标/线条/阴影等全部视觉参数。目标：新增页面除文案外所有视觉元素与当前系统一模一样。规范输出到 `docs/design/spec.md`
     - ❌ 暂不整理 → 进入第三步

   **第三步：是否有参考页面**
   - 询问用户：是否有参考页面或同类产品界面？
     - ✅ 有 → 按参考风格设计，提取参考页面的色彩/字体/布局/间距特征
     - ❌ 无 → `designer` 自主设计，但需先向用户确认设计方向（风格/色彩/响应式）

   **设计约束（无论走哪条路径都必须遵守）：**
   - 🔄 **控件复用**：优先使用项目已有的前端组件库中的控件，避免重复造轮子
   - 🎨 **风格统一**：新设计必须与当前系统风格保持一致（色彩/字体/间距/圆角/阴影）
   - 🧩 **架构统一**：遵循项目已有的前端架构模式（组件结构/状态管理/路由/布局）

   **设计完成后自检：与当前系统页面逐项对比**：
   - 新设计稿与已有页面并排对比（截图对照），逐项检查：字体字号字重/按钮尺寸圆角颜色/表格表头行高边框/表单label输入框/弹框圆角阴影按钮/间距gap/padding/分割线颜色粗细/图标库尺寸
   - 发现差异 → 修改设计使其与现有系统一致后再进入评审
   - 确认无差异 → 进入设计评审
6. **设计评审** — 联合会议评审，`designer` 主持，全体参与：
   - `designer`（UI/UX）— 主持 + 设计规范一致性
   - `analyst`（产品）— 需求覆盖度与优先级对齐
   - `executor`（前端）— 前端实现可行性与组件复用
   - `architect`（后端）— API/数据接口对齐
   - `test-engineer`（测试）— 交互状态覆盖 + 无障碍 WCAG 2.2 AA 检查
   - 各角色各自输出评审意见，设计师汇总。无法达成一致的提交用户决策
7. **用户确认设计** — 🚨 **硬性关卡**：提交设计稿供用户确认。未确认前**禁止**进入任何后续阶段（含测试准备和开发）。如从 hlpm 进入且设计未变更，可跳过

### 第二阶段：测试准备
8. **测试用例编写** — `test-engineer` 编写测试用例（含E2E场景）
9. **用例评审** — `test-engineer` + `analyst` 联合评审用例

### 第三阶段：开发阶段
10. **Git 工作区准备** — 创建隔离工作区进行开发，避免污染主工作区：
    - 检测当前环境（普通仓库 / 工作区）
    - 如在主分支，先创建 feature 工作区分支
    - 安装依赖并验证干净基线（运行现有测试确保通过）
11. **开发计划** — `planner` 制定开发计划并按 [任务分配模板] 拆分任务。每个任务满足微任务粒度：**单一步骤（2-5分钟可完成）**，含精确文件路径、完整代码、具体命令，无占位符和"TBD"。任务执行遵循受阻停止纪律
12. **架构决策记录** — 对关键技术选型以 ADR 格式记录到 `docs/adr/NNNN-<slug>.md`（4 位编号，详见 `hladr`）
13. **开发** — `executor`（前端+后端）开发，遵循技术设计文档、编码标准、API设计规范、数据库迁移规范、错误处理规范、无障碍规范。推荐使用 TDD，覆盖率 >= 80%
    - **并行策略**：前端和后端任务如无依赖关系，可并行分派执行
    - 多个独立模块的任务可同时分派给多个 `executor` 实例并行推进
14. **代码审查** — `code-reviewer` 审查代码质量，按严重/重要/次要分级输出
15. **前后端联调** — 前后端对接接口，联合调试
16. **前后端自测** — 开发自测功能完整性与接口正确性
17. **分支完成** — 代码审查通过后执行分支收尾：检测环境→验证测试通过→确定基础分支→呈现4选项（本地合并/推送PR/保持原样/丢弃）→执行选择→清理工作区

### 第四阶段：测试
18. **安全审查** — `security-reviewer` 执行安全漏洞扫描
19. **测试用例测试** — `qa-tester` 按用例执行测试（单元+集成+E2E）
20. **浏览器验证测试** — `qa-tester` 浏览器环境验证，含无障碍测试
21. **生产审计** — `verifier` 执行结构化审计，输出风险评分（0-100）。评分 < 50 受阻不可发布

### 第五阶段：交付与发布
22. **交付验证** — `verifier` 验收所有文档与功能完整性
23. **交付** — 产出全部项目文档，保存到项目目录：
    - `docs/` → 需求文档/PRD/技术设计/ADR/测试用例/回滚方案
    - `CHANGELOG.md`（项目根目录特例，业界惯例）→ 变更日志
    - `docs/design/` → 设计规范文档/设计HTML
    - `docs/user/` → 用户操作手册/帮助文档
    - 🚨 如不确定项目根目录，向用户提问确认
24. **用户确认发布** — 🚨 **硬性关卡**：向用户展示版本号/变更日志/预发布清单，用户确认后方可推送标签。严禁自动发布
25. **发布** — 用户确认后执行标签推送（版本号→变更日志→标签→推送）
26. **用户确认部署** — 🚨 **硬性关卡**：向用户展示部署策略/目标环境/版本号/回滚方案，用户确认后方可部署。严禁自动部署
27. **部署** — 用户确认后执行部署（CI/CD→部署策略→健康检查→回滚）

## 问题升级机制

- 各阶段遇到问题，由**产品经理**（`analyst`）组织会议讨论
- 会议中无法达成一致的问题，**立即向用户提问**，不得自行裁决
- 直至所有参与方无异议后，方可进入下一阶段

## 变更传播规则

当用户在"确认设计页面"阶段提出修改时，**所有关联文档必须同步更新**，产品经理负责追踪变更。

## 交付文档清单

| # | 文档 | 负责 Agent | 说明 |
|---|------|-----------|------|
| 1 | 需求文档 | `analyst` | 原始需求描述与业务背景 |
| 2 | 竞品分析报告 | `analyst` | 新项目必须，竞品识别/功能对比/差异化定位 |
| 3 | PRD文档 | `analyst` | 产品需求详细规格说明 |
| 4 | 设计规范文档 | `designer` | 设计系统/组件库/色彩/字体/间距规范 |
| 5 | 设计HTML | `designer` | 可预览的交互设计稿 |
| 6 | 技术设计文档 | `architect` | 接口文档 + 数据库设计 + 逻辑架构图 |
| 7 | 架构决策记录 | `architect` | ADR 格式记录到 `docs/adr/NNNN-<slug>.md` |
| 8 | 测试用例 | `test-engineer` | 功能用例/边界条件/异常场景/E2E |
| 9 | **用户操作手册** | `writer` | 如为终端用户系统，含截图与操作视频 |
| 10 | 帮助文档 | `writer` | 常见问题/故障排查 |
| 11 | 变更日志 | `planner` | 从 Git 提交历史生成 CHANGELOG |
| 12 | 回滚方案 | `architect` | 本次发布的回滚步骤与验证方法 |

### 用户操作手册生成指引
`writer` 编写用户操作手册时：
- 📸 截图标注：优先用模型原生多模态分析页面截图，标注操作步骤序号和箭头
- 🎬 操作视频：可选，使用可选外部 `videodb` 技能录制或处理演示视频（需用户自行安装）
- 📖 结构：快速入门 → 核心功能操作 → 常见问题 → 故障排查
- 🧪 验证：每条操作步骤必须实测可复现
