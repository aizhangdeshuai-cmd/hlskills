# hlpm 使用说明

> **多角色协作的产品段(从需求到交付物)**
> 完整规范见 [`SKILL.md`](./SKILL.md),本 README 聚焦"怎么用"

---

## 这是什么

`hlpm` 是 hlskills 体系中**多角色协作(产品+开发)流程的"产品段"**。
当你的团队是"产品经理 + 开发"两个角色分工时,用这个技能。

**它做什么**:接收你的需求 → 输出 8 类交付物 → 交给 `hldev` 开发
**它不做什么**:不改任何代码(改代码请用 `hldev`)

---

## 什么时候用

✅ **适合用 hlpm**:
- 你负责产品,开发同事负责代码
- 需求从你这里发起,需要先评审再交接
- 团队有"评审会签"机制

❌ **不适合**(改用其他技能):
- 你一个人跑全流程 → 用 `hl-flow`
- 你要直接改代码 → 用 `hldev`
- 你要复盘/重构已有项目 → 用 `hllegacy` / `hlrefactor`

```
我有需求,接下来怎么办?
│
├─ 只有我一个人,产品+开发都是我
│  └─ 👉 用 hl-flow(28 步单人全流程)
│
├─ 我有产品+开发分工,需求从我这发起
│  │
│  ├─ 想一次跑完(产品段→开发段→发布→部署)
│  │  └─ 👉 用 hlchain(6 阶段编排,自动调 hlpm/hldev/hlrelease/hldeploy)
│  │
│  └─ 想分阶段控制(先产品段做评审,再交接给开发)
│     └─ 👉 用 hlpm → 走完 → 交接给 hldev(分两步)
│
├─ 只想做评审(已有 PRD/设计,看哪里有问题)
│  └─ 👉 用 hlreview(独立评审,不写产物)
│
├─ 紧急 Bug 修复/线上排查
│  └─ 👉 用 hlbug(14 步快速定位修复)
│
└─ 已有项目想重构/分析
   ├─ 梳理项目现状/设计规范 → hllegacy
   └─ 计划重构 → hlrefactor
```

> **关键判断点**: 你能独立完成开发吗?
> - **能** → hl-flow(单人 28 步)
> - **不能,需要开发同事** → hlpm + hldev(产品段 + 开发段分工)或 hlchain(全链路一次跑完)

## 5 步快速上手

### 第 1 步:启动技能

```
/hlpm
```

或自然语言:`我有个新需求,启动产品段流程`

启动后 AI 会自动执行**启动检查清单**(详见 SKILL.md 顶部):
- 确认是产品段(非开发段)
- 询问需求规模
- 询问是否涉及设计
- 询问评审模式

### 第 2 步:回答 3 个问题(0.5 步)

| 问题 | 选项 | 影响 |
|------|------|------|
| **Q1 需求规模** | 轻量 / 标准 / 复杂 | 决定走多少步骤 |
| **Q2 涉及设计** | 涉及 / 不涉及 | 决定是否出设计稿 |
| **Q3 评审模式** | 分阶段评审 / 集中评审 | 决定走 3 场评审还是 1 场集中评审 |

**不知道怎么选?**:
- 改 1 个按钮/字段 → 轻量 + 集中评审
- 改 1 个页面/功能模块 → 标准 + 分阶段评审
- 跨多模块/新项目 → 复杂 + 分阶段评审(强制)

### 第 3 步:AI 帮你跑流程

AI 会**自动执行**以下步骤(你只需要在阻塞点回答问题):

```
加载项目上下文 → 选版本目录 → 需求分析 → 竞品分析(条件) →
PRD 编写 → 评审 → 设计(条件) → 用例编写 → 集中评审/3 场评审 →
一致性终检 → 验收标准 → 自检报告 → 打包交付
```

**关键节点**(AI 会主动停下来问你):
- 2a:有参考竞品吗?
- 0.6:有历史未开发版本,要在旧版本上改还是新建?
- 5/7/9 或 9a:评审会签(轻量走 9a 集中评审)
- 强同步触发时:哪几项要重走

### 第 4 步:查看产出物

完成后,所有交付物在 `docs/v{ver}/` 目录:

```
docs/v1/
├── prd.md                            ← PRD(必出)
├── test-cases.md                     ← 测试用例(必出)
├── acceptance-criteria.md            ← 验收标准(必出)
├── non-functional-requirements.md    ← 非功能需求(必出)
├── consistency-matrix.md             ← 一致性矩阵(必出)
├── handoff-self-check.md             ← 自检报告(必出)
├── design/
│   └── *.html                        ← 设计稿(条件出)
└── analysis/
    └── competitive-analysis.md        ← 竞品分析(条件出)
```

`{ver}` 是 0.6 步自动决定的版本号(`v1` / `v2` / `v3` ...)。

### 第 5 步:通知开发接手

把 `docs/v{ver}/` 整个目录路径告诉开发同事(或直接说"启动 hldev")。

**开发接手会做**:
- 验证 14 项交付物齐全 (8 项产品段 + 6 项开发段补充)
- 验证版本号一致(PRD v1 = 设计 v1 = 用例 v1)
- 验证一致性矩阵全部 ✅
- 验证自检报告签字方完整
- **任一不合规 → 拒收 + 给你拒收原因**(详见 `hldev/handoff-rejection.md`)

## 6 种路径(按你的回答)

| 规模 | 涉及设计 | 评审模式 | 路径 | 跳过步骤 | 适合 |
|------|---------|---------|------|---------|------|
| 轻量 | 涉及 | 集中评审 | 轻量+设计+集中评审 | 竞品/中间 3 场评审/部分矩阵 | 1 个页面加 1 个功能 |
| 轻量 | 不涉及 | 集中评审 | 轻量无设计+集中评审 | 竞品/整个第二阶段/部分矩阵 | 加 1 个接口/字段 |
| 标准 | 涉及 | 分阶段评审(默认) | 完整 23 步 | (无跳过) | 加 1 个功能模块 |
| 标准 | 不涉及 | 分阶段评审(默认) | 标准无设计 | 整个第二阶段 | 改后端逻辑 |
| 复杂 | 涉及 | 分阶段评审(**强制**) | 完整 23 步 + 加严 | (无跳过) | 跨模块新项目 |
| 复杂 | 不涉及 | 分阶段评审(**强制**) | 复杂无设计 | 整个第二阶段;非功能加深 | 多模块后端重构 |

## 关键概念

### 评审模式

| 模式 | 评审场次 | 适合 | 不通过时 |
|------|---------|------|---------|
| **分阶段评审** | 3 场(PRD/设计/用例各 1 场,5 方会签) | 标准/复杂需求 | 驳回 → 强制重走后续所有评审 |
| **集中评审** | 1 场(PRD/设计/用例**全部写完后**开 1 次会, 5 角色集中讨论) | 轻量需求 | 驳回 → 改 → 重走 1 场;1 轮不通过升级为分阶段评审 |

**集中评审 ≠ "少角色"**: 仍是 5 角色 (`analyst` 主持 + `architect` + `designer` + `executor` + `test-engineer`), 只是从"开 3 次会"压缩为"开 1 次会", 速度更快但**保持交叉审视**。

**强同步规则**:任一评审驳回后,后续**所有评审必须重走**(不能豁免已通过的)。
- 例:设计评审被驳回 → 即使 PRD/用例原本已通过,也要重走 PRD 评审 + 用例评审

### 版本目录

每次产品段流程产出到独立目录 `docs/v{N}/`,不覆盖历史。

**启动时会自动**:
- 扫描历史版本(看 `.dev-completed` 标记文件)
- 如果有"未开发"的旧版本 → 问你"在旧版本上改 / 新建版本"
- 全部已开发 → 自动建新版本(如 `v3`)

**好处**:不会丢失历史;可对比各版本差异;开发拒收后产品可在原版本补齐。

### 设计规范遵循(硬性)

**如果你项目里有设计规范**(`docs/design/spec.md`):
- AI 会读取规范
- 设计稿必须**字面遵循**(颜色/字号/间距值都按规范,不可"参考发挥")
- 每条设计决策都要在 HTML/CSS 注释里引用规范条款(如 `/* 规范: §3.2 圆角值 */`)

**如果项目里没有规范**:
- AI 会输出"未发现设计规范"报告
- 你可以选择:先用 `hllegacy` 整理规范 / 直接开始设计

### 已有页面微调

**如果你要在已有页面上改**:
- AI 会先问"全新页面 / 已有页面微调"
- 选"已有页面微调" → AI 会读原页面代码 → 输出**3 区块 diff 设计稿**:
  - 区块 1: 原页面代码(diff 基线)
  - 区块 2: 改动清单(每条改动 + 原因 + 规范引用 + 影响范围)
  - 区块 3: 修改后完整页面(diff 标记)

文件名用 `.diff.html` 后缀(如 `order-list.diff.html`)。

## 常见问题

### Q1:为什么我不能直接改代码?

**A**:本技能是"产品段",职责是"定义做什么",不负责"怎么实现"。改代码是 `hldev` 的职责。这样分工的好处:
- 产品专注于业务规则
- 开发专注于实现质量
- 角色边界清晰,可拒收可追溯

即使只是"改一行字段名",也建议走 `hldev`,因为:
- dev 会跑测试,确保改动不破坏现有功能
- dev 会做代码审查
- dev 会在版本控制里留完整记录

### Q2:小需求也要走完整流程吗?

**A**:不一定。按 0.5 步骤选择"轻量 + 集中评审",会跳过:
- 竞品分析
- 中间 3 场重量评审(合并为 1 场集中评审)
- 一致性矩阵的部分矩阵(免跑权限/非功能)

**轻量需求实际只跑 12 步**(对比完整 23 步)。

### Q3:评审被驳回后会发生什么?

**A**:
- 单项评审驳回 → 返回对应步骤修改 → 重走该评审(最多 3 轮)
- 3 轮后仍不通过 → 升级到用户决策(让你判断)
- 触发强同步 → 后续已通过的评审也必须重走(防止"只改一处,其他地方不一致")

### Q4:如果开发拒收怎么办?

**A**:开发会填写 `handoff-rejection.md`(拒收单)给你,**说明拒收原因**(可能是缺文档/版本不一致/内部矛盾等)。

你需要:
1. 看拒收原因
2. 回到产品段补齐(可能回到步骤 4/6b/8/11)
3. 重走对应评审
4. 重新打包交付

**完整拒收流程**见 [`hldev/handoff-rejection.md`](../hldev/handoff-rejection.md)。

### Q5:可以同时跑多个需求吗?

**A**:可以,但**每个需求走独立版本目录**:
- 需求 A → 走 `docs/v1/`
- 需求 B → 走 `docs/v2/`
- 需求 C → 走 `docs/v3/`

需求之间互不干扰,可同时进行(AI 会按版本号区分)。

### Q6:设计规范在哪里维护?

**A**:项目级规范放在:
- 优先 `docs/design/spec.md`
- 备选 `docs/design/SKILL.md` / `docs/design/*.md`
- 兜底 项目根目录 `DESIGN.md`

如果没有规范,**先调用 `hllegacy` 第 11 步整理规范**,再启动产品段。

### Q7:我应该用 hlpm 还是 hlpm?

| 场景 | 用哪个 |
|------|-------|
| 团队只有你一个人(产品+开发都你做) | `hl-flow` |
| 团队有产品+开发分工 | `hlpm` + `hldev` |
| 已有项目,要做小改动 | `hlpm` + `hldev`(轻量 + 集中评审) |
| 全新项目 0→1 | `hlpm` + `hldev` (产品段产 PRD/用例,再 hldev 接手) |

**`hl-flow` (单人跑全流程) / `hlpm` (产品段多角色) / `hldev` (开发段多角色) 三件套, 按需选用**。
