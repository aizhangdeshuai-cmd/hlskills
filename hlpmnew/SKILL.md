> 如发现 Agent 在第 X 步没停下来等你确认 / 直接跳过第 Y 步 / 没返回上一步重做,请**手动打断**让它重读 SKILL.md 对应章节。

## 开发流程

### 第一阶段：市场与可行性（6 步）
> ⚠️ 本阶段所有信息缺口必须向用户提问，不可自行假设。
1. **市场调研** — `analyst` 调研市场规模/趋势/用户痛点，如用户提供参考文件（含图片）则读取图片内容
2. **竞品分析** — `analyst` 分析 3-5 个竞品：功能矩阵/定价/优劣势，输出竞品分析报告
3. **目标用户画像** — `analyst` 输出用户群体/使用场景/核心诉求排序
4. **差异化定位** — `analyst` 明确独特优势，不能一句话说清则标注 🔴 风险
5. **技术可行性** — `architect` 评估技术栈/关键难点/第三方依赖
6. **资源与排期** — `planner` 估算人力/技能/时间线

### 第二阶段：风险评估与决策（3 步）
> ⚠️ 本阶段所有信息缺口必须向用户提问，不可自行假设。
7. **风险识别** — `analyst` + `architect` 识别技术/市场/资源/合规/依赖风险
8. **风险分级** — 每个风险标定严重性（🔴/🟡/🟢）+ 概率（高/中/低）+ 应对预案
9. **Go/No-Go 决策** — 向用户呈现分析结论：市场规模/竞品定位/技术可行性/资源需求/风险。🚨 用户决策：Go（继续）/ No-Go（终止）

### 第三阶段：需求与设计（8 步）
> ⚠️ 本阶段所有信息缺口必须向用户提问，不可自行假设。风格方向、功能边界、交互细节等不确定就开口问。
10. **项目章程** — `planner` 起草：目标/范围/干系人/成功标准/约束
11. **需求文档** — `analyst` 生成核心功能范围与非功能需求内容 → 写入文件，图片一并分析
12. **PRD 编写** — `analyst` 生成完整详细的产品需求文档内容 → 写入文件。**下游角色（设计师、开发、测试）阅读 PRD 后不得有疑问或猜测**：
    - **业务逻辑**：每个功能的完整业务规则，包含所有分支条件、边界情况、异常路径。例如"用户下单"不只是"选商品→付款"，要覆盖：库存不足、优惠券叠加规则、运费计算、支付超时处理、取消订单的退款路径等
    - **操作流程**：每个用户操作的主流程+替代流程+异常流程，用流程图或编号步骤描述。覆盖从进入页面到离开页面的完整路径
    - **数据流转**：数据从哪里来（输入）、经过哪些处理（逻辑）、到哪里去（输出/存储）。字段级别的数据定义（字段名/类型/必填/校验规则/默认值/示例值）
    - **状态机**：核心实体（订单/用户/任务等）的完整状态流转图，每个状态下可执行的操作
    - **权限规则**：每个功能/页面的访问权限（游客/登录用户/管理员等）
    - **非功能需求**：性能指标（响应时间/QPS）、安全要求、兼容性要求
    - ⚠️ **不确定就问**：任何业务逻辑、数据规则、流程细节有不确定的地方，**立即向用户提问**，不假设、不猜测、不留空。PRD 中出现"暂定""待确认""或""等"等模糊字眼前必须先问过用户
    - ✅ **自检标准**：PRD 写完后，随机抽取 3 个功能点，模拟设计师/开发/测试角色分别提问——如果任何一个角色会问"这个情况怎么处理？"，PRD 不合格，补齐后再进入评审
13. **PRD 评审** — 联合会议评审，`analyst` 主持，全体参与：
    - `analyst`（产品）— 需求完整性与优先级
    - `architect`（后端）— 技术可行性与架构影响
    - `executor`（前端）— UI 可实现性与交互合理性
    - `designer`（UI/UX）— 设计可行性与用户体验
    - `test-engineer`（测试）— 可测试性与边界条件覆盖
14. **整体风格设计** — `designer` 先定风格基调，不画具体页面。输出内容包括：
    - **配色方案**：主色/辅色/强调色/中性色色板，亮色+暗色两套
    - **字体系统**：标题字体/正文字体/代码字体，各级字号与行高比例
    - **间距系统**：基础单位（4px/8px），各级间距（xs/sm/md/lg/xl）
    - **圆角与阴影**：按钮/卡片/弹窗的圆角半径，阴影层级（1-4 级）
    - **核心组件样式**：按钮（主/次/危险/禁用 4 态）、输入框（默认/聚焦/错误/禁用 4 态）、标签/徽章/头像
    - **动效基调**：hover 反馈、过渡时间、缓动曲线
    - 输出格式：可预览的 HTML 风格指南页面（含所有色板/字体/组件示例）
    - 🚨 **必须产出 HTML 预览**：`designer` 生成独立 HTML 文件展示完整风格系统，用户可在浏览器中实际看到效果
15. **用户确认风格** — 🚨 **硬性关卡**：
    - 向用户展示风格预览 HTML + 设计意图说明（为什么选这些配色/字体）
    - 用户确认整体方向，如需调整则回到步骤 14 迭代
    - 风格未确认之前，**禁止进入页面设计**
    - 提供对比选项（如"这是两个配色方向，你更倾向哪个？"）帮助用户决策
16. **全部页面详细设计** — `designer` 风格确认后，逐页面详细设计：
    - 覆盖 **所有页面**：列表页、详情页、表单页、设置页、登录/注册页、404/错误页、空状态页、加载态
    - 每个页面覆盖 **全部交互状态**：默认态、hover 态、空数据态、加载态、错误态、成功态、极限边界态（最长文本/最多列表项）
    - 每个页面输出可预览 HTML，包含真实内容（非 Lorem ipsum，用贴近真实场景的示例数据）
    - 页面间导航流已串联，可在浏览器中逐页浏览
    - 设计完成后的自检清单：
      - ✅ PRD 中每个功能都有对应页面
      - ✅ 每个页面覆盖了正常/空/错/加载/边界 5 种状态
      - ✅ 所有可点击元素有 hover/focus/active 样式
      - ✅ 移动/平板/桌面三屏布局均正常
      - ✅ 字体/颜色/间距/圆角与风格指南一致
17. **设计评审** — 联合会议评审，`designer` 主持，全体参与：
    - `designer`（UI/UX）— 设计规范一致性
    - `analyst`（产品）— 需求覆盖度
    - `executor`（前端）— 实现可行性与组件复用
    - `architect`（后端）— API/数据接口对齐
    - `test-engineer`（测试）— 交互状态覆盖 + 无障碍检查

### 第四阶段：测试准备（5 步）
18. **技术设计骨架** — `architect` 先搭骨架，不追求一次性完整。输出：
    - **接口文档骨架**：核心 API 端点清单（方法+路径+用途一句话），请求/响应结构在开发中补全
    - **数据库设计骨架**：核心表清单（表名+用途一句话），字段细节在开发中补全
    - **逻辑架构图**：模块划分与依赖关系，可在开发中调整
    - ⚠️ **关键原则**：骨架足够指导开发起步即可，细节在第五阶段开发中持续补全，**开发结束 = 文档最新**
19. **架构决策记录** — `architect` 对关键技术选型记录 ADR 到 `docs/adr/NNNN-<slug>.md`（4 位编号，详见 `hladr`）
20. **测试用例编写** — `test-engineer` 编写测试用例（含 E2E 场景）
21. **用例评审** — `test-engineer` + `analyst` 联合评审
22. **用户确认设计** — 🚨 **硬性关卡**：提交设计稿供用户确认，未确认不得进入开发

### 第五阶段：开发（8 步）
23. **Git 工作区准备** — 创建隔离工作区，安装依赖，验证干净基线
24. **开发计划** — `planner` 按微任务粒度拆分任务（2-5 分钟/任务），计划中明确哪些任务涉及文档补全
25. **开发** — `executor`（前端+后端）并行开发，遵循所有技术规范。**边开发边完善技术文档**：
    - 每完成一个 API 端点 → 立即补全接口文档的请求/响应结构
    - 每创建/修改一个表 → 立即更新数据库设计的字段定义
    - 架构有调整 → 立即更新逻辑架构图
    - 不做"开发完再补文档"，文档与代码同步演进
26. **代码审查** — `code-reviewer` 审查代码质量，严重/重要/次要分级。**同步检查**：改动涉及的接口文档/数据库设计是否已更新
27. **前后端联调** — 接口对接，联合调试。联调中发现接口变更 → 同步更新接口文档
28. **前后端自测** — 开发自测功能完整性与接口正确性
29. **分支完成** — 代码审查通过后执行分支收尾（合并/PR/保留/丢弃）
30. **用户确认开发结果** — 🚨 向用户展示开发成果，确认后方可进入测试。**展示时包含**：
    - 功能演示
    - 最新技术文档（接口文档 + 数据库设计 + 架构图）— 应与当前代码完全一致

### 第六阶段：测试（5 步）
31. **安全审查** — `security-reviewer` 安全漏洞扫描
32. **测试用例测试** — `qa-tester` 按用例执行（单元+集成+E2E）
33. **浏览器验证** — `qa-tester` 浏览器环境验证，含无障碍测试、响应式三屏
34. **健康评分** — 8 类加权评分（功能/控制台/A11y/UX/视觉/性能/链接/内容），< 50 阻断发布
35. **生产审计** — `verifier` 结构化审计，评分 < 50 受阻不可发布

### 第七阶段：交付与上线（6 步）
36. **文档生成** — 各 Agent 按分工生成交付文档，保存到项目目录（`docs/` / `docs/design/` / `docs/analysis/` / `docs/user/`）。不确定项目根目录则向用户提问确认
37. **交付验证** — `verifier` 验收所有文档与功能完整性
38. **用户确认发布** — 🚨 **硬性关卡**：展示版本号/变更日志/预发布清单/健康评分，用户确认后方可推送
39. **发布** — 用户确认后：版本号→变更日志→标签→推送
40. **用户确认部署** — 🚨 **硬性关卡**：展示部署策略/目标环境/版本号/回滚方案，用户确认后方可部署
41. **部署** — 用户确认后：CI/CD→部署策略→健康检查→回滚

## 问题升级机制

- 各阶段由产品经理 `analyst` 组织会议
- 无法达成一致立即向用户提问
- 所有人无异议方可进入下一阶段

## 变更传播规则

- 设计变更 → 需求/PRD/设计规范/设计HTML/测试用例/技术设计 **同步更新**
- 开发中 API/数据库/架构调整 → 技术文档 **同步更新**（不等开发结束再补）
- 产品经理负责追踪，确保一致性
