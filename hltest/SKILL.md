# 测试全流程规范

> 属于 `hlskills` 技能系统。融合 GStack QA 的注释截图、健康评分、Diff-aware 模式。

## 角色

| 角色 | Agent |
|------|-------|
| 测试用例 | `test-engineer` |
| 测试执行 | `qa-tester` |

## 测试流程

### 测试用例编写
`test-engineer` 编写测试用例，覆盖：
- 功能测试用例（核心功能路径）
- 边界条件（输入边界、边界值）
- 异常场景（错误处理、网络异常、超时）
- E2E 场景（用户核心旅程）

### 用例评审
`test-engineer` + `analyst` 联合评审，输出：通过 / 修改建议 / 驳回

### 测试执行
`qa-tester` 按用例执行：单元测试 + 集成测试 + E2E测试

### 浏览器验证
`qa-tester` 浏览器环境最终验证，含无障碍测试：
- 屏幕阅读器测试（VoiceOver/NVDA）
- 键盘导航全流程测试
- 焦点陷阱检查
- **视觉回归**：截图对比重构前后/开发前后，使用 Read 工具读取截图对比差异
- **注释截图**：在关键交互元素上标注引用标签，红色边框叠加输出清晰错误证据
- **动作前后差异**：click/fill 前截一张、后截一张，对比验证操作是否生效

## 通用质量规则（强制覆盖维度）

> **每个操作按钮 / 每个页面 / 每个功能点,必须覆盖以下通用测试维度**。
> 这部分独立于 PRD 业务规则,无论 PRD 是否提及,均需生成对应用例。

### 规则清单（10 项必含）

| # | 维度 | 触发条件 | 至少产出用例数 |
|---|------|----------|:----------:|
| 1 | **防重复点击** | 每个"新增/提交/执行/保存/删除/确认"按钮 | 1 条 |
| 2 | **Tab 切换数据一致性** | 每个含 Tab 的页面 | 1 条 |
| 3 | **逆向业务操作** | 每个功能点(新增必有删除,提交必有撤回) | 按功能点 |
| 4 | **软删除过滤** | 涉及删除的场景 | 1 条 |
| 5 | **输入长度边界** | 每个输入框(最小/最大/超长) | 按字段 |
| 6 | **特殊字符 / XSS** | 任何接受用户输入的字段 | 1 条 |
| 7 | **筛选条件** | 每个列表页 | 按筛选项 |
| 8 | **列表分页** | 每个列表页 | 1 条 |
| 9 | **字段值展示** | 列表 / 详情页字段 | 按字段 |
| 10 | **添加多条明细** | 涉及子表/明细行的页面 | 1 条 |

### 各维度详述

#### 1. 防重复点击
- **场景**:用户快速连点提交按钮 5 次
- **预期**:只产生 1 条业务数据(或后端幂等返回同一结果),按钮 loading 状态锁住

#### 2. Tab 切换数据一致性
- **场景**:Tab A 切换到 Tab B 再切回 Tab A,或快速交替点击 5 次
- **预期**:列表数据、筛选条件、分页位置保持一致(或按设计重置,需文档化)

#### 3. 逆向业务操作
- **正向**:新增 → 编辑 → 删除
- **逆向**:删除后能否恢复?编辑后能否撤销?提交后能否撤回?
- **每对操作至少 1 条逆向用例**

#### 4. 软删除过滤
- **场景**:删除一条数据后,查询/统计/导出/列表默认视图
- **预期**:软删数据不出现;打开"显示已删除"开关才可见;关联数据(子表/外键)状态正确

#### 5. 输入长度边界
- **每个输入框**生成 3 条:
  - 最小长度(0 字符 / 1 字符)
  - 正常长度
  - 超长(超过 maxlength + 1)
- **预期**:`maxlength` 截断 / 校验提示 / DB 字段长度限制

#### 6. 特殊字符 / XSS
- **场景**:输入 `<script>alert(1)</script>` / `&"/'><` / SQL 注入 `' OR 1=1--`
- **预期**:前端展示已 HTML 实体转义;后端参数化查询无注入;存储原文不解析

#### 7. 筛选条件
- **每个筛选项**(下拉/输入/日期/枚举)至少 1 条:
  - 有效值命中
  - 空值(全部)
  - 边界值(首项/末项)
  - 组合筛选(多条件 AND/OR)

#### 8. 列表分页
- **场景**:第 1 页 / 末页 / 跨页操作(删除第 2 页最后一条后跳第 1 页) / 切换每页条数
- **预期**:分页准确,无重复/遗漏,总数与列表一致

#### 9. 字段值展示
- **每个展示字段**:
  - 正常值
  - 空值(显示 "—" 或空字符串)
  - 超长(截断 + 悬停气泡)
  - 特殊值(0 / null / 负数 / 日期格式)

#### 10. 添加多条明细
- **场景**:连续添加 50 条 / 100 条明细行
- **预期**:性能可接受(响应 < 2s);无重复行号;保存完整无遗漏;翻页/虚拟滚动正常

### 业务专业性注入
- 调用方若提供行业提示词(船舶/医疗/金融等),必须结合行业知识**丰富用例数量与深度**
- 例如船舶维修场景:航行中/靠港中/停航中等特殊状态下的操作限制

### 与 PRD 规则的关系

```
PRD 规则(纵向深挖)        +        通用质量规则(横向广覆盖)
       ↓                                    ↓
业务正确性用例              +        健壮性/容错/UI 用例
       ↓                                    ↓
                    完整测试用例集
```

- **PRD 规则驱动** — 覆盖业务正确性(做且仅做 PRD 约定的)
- **通用质量规则驱动** — 覆盖健壮性(防重/容错/边界)
- **两者缺一不可**

### 输出要求
- 每条用例必须标注属于哪类规则:`[PRD]` / `[防重]` / `[Tab]` / `[逆向]` / `[软删]` / `[长度]` / `[XSS]` / `[筛选]` / `[分页]` / `[字段]` / `[明细]`
- 用例总数建议:通用质量规则部分不少于 PRD 业务规则的 50%
- 覆盖矩阵(第 8 节)需同时列出 PRD 规则覆盖和通用质量规则覆盖

## QA 层级（融合 gstack /qa）

QA 执行分三个层级，根据场景选择：

| 层级 | 耗时 | 覆盖范围 | 适用场景 |
|------|:---:|------|------|
| **Quick** | ~30s | 首页 + Top 5 导航目标冒烟 | PR 合并前、小改动验证 |
| **Standard** | ~5min | 核心用户路径 + 表单提交 + 数据展示 + 响应式三屏 | 功能分支完成、常规发布前 |
| **Exhaustive** | ~30min | 所有可达页面 + 全交互元素 + 边界条件 + 无障碍审计 | 大版本发布、首次上线、安全审计 |

**输出要求**：
- 每次 QA 必须输出**健康评分**（0-100）和**问题清单**（标题 + 严重性 + 复现步骤 + 注释截图证据）
- 修复后重新运行 QA，产出**前后健康评分对比**
- 使用 `hlbrowse` 的 `snapshot -a` 注释截图作为证据标准

**仅报告模式（融合 gstack /qa-only）**：
- 产出完整 Bug 报告但不修复，适合独立测试人员或用户验收测试
- 报告保存到 `docs/qa/`，格式：问题标题 + 严重性（Critical/Major/Minor）+ 复现步骤 + 预期 vs 实际 + 截图证据

## 测试模式

### 模式选择

| 模式 | 触发 | 做什么 |
|------|------|--------|
| **Diff-aware** | 在功能分支上，有 git diff | 分析 `git diff main...HEAD`，识别受影响页面/路由，精准测试变更范围 |
| **Full** | 提供完整 URL | 系统探索所有可达页面，记录 5-10 个问题，生成健康评分 |
| **Quick** | --quick | 30 秒冒烟：首页 + Top 5 导航目标 |
| **Regression** | 对照基线 | 运行 Full 模式，加载之前 baseline.json，区分新旧问题 |

### Diff-aware 模式（默认）

当在功能分支上执行测试时，默认启用：
1. 运行 `git diff main...HEAD` 识别变更文件
2. 映射变更文件到受影响页面/路由
3. 检查本地服务端口（:3000、:4000、:8080 等）
4. **只测试受影响的页面**，不浪费时间在无关页面
5. 如未检测到本地服务，提示用户启动

## 健康评分系统

每次完整测试后输出加权健康评分（0-100）：

| 类别 | 权重 | 检查项 |
|------|:---:|------|
| 功能正确性 | 20% | 核心路径是否正常、表单提交、数据展示 |
| 控制台错误 | 15% | console.error 数量与严重性 |
| 无障碍 | 15% | WCAG 2.2 AA 合规度 |
| 用户体验 | 15% | 交互流畅度、加载态/空态/错误态处理 |
| 视觉一致性 | 10% | 与设计稿对比、响应式适配 |
| 性能 | 10% | 首屏加载时间、LCP、CLS |
| 链接有效性 | 10% | 死链检测、外链可达性 |
| 内容完整性 | 5% | 文案正确、无缺失、无乱码 |

### 评分标准
- **85-100** 强 — 可发布
- **70-84** 可发布 — 有少量问题需跟进
- **50-69** 有风险 — 建议修复后再发布
- **< 50** 受阻 — 不可发布，必须修复关键问题

### 健康评分消费方
以下技能在决策时必须读取健康评分：
- `hlpm` 第 21 步（生产审计）：健康评分 < 50 → 阻断发布
- `hlbug` 第 13 步（最终验证）：健康评分低于修复前 → 回退修复
- `hlrefactor` 第 19 步（生产审计）：健康评分低于重构前基线 → 阻断合并

## 响应式多屏测试

每次浏览器验证须覆盖三个视口：

| 视口 | 分辨率 | 用途 |
|------|--------|------|
| 移动端 | 375×812 | iPhone 尺寸 |
| 平板 | 768×1024 | iPad 尺寸 |
| 桌面端 | 1280×720 | 标准桌面 |

- 每个视口截取关键页面全屏截图
- 检查布局是否正常（无溢出、无重叠、无截断）
- 交互元素目标尺寸 >= 24×24px（移动端必查）

## E2E 测试规范

遵循 `e2e-testing` 技能，核心规则：

### 工具与架构
- 使用 **Playwright** 作为 E2E 测试框架
- 页面交互使用 **Page Object Model**：定位器、动作、导航分离

### 配置
```
完全并行运行
失败自动重试 2 次
超时：action 10s / navigation 30s / global 60s
```

### 产物管理
- 失败时保留截图（含注释截图）
- 首次重试时保留追踪
- 录制视频

### 不稳定性策略
- 不稳定测试隔离到单独文件，不阻塞主测试套件
- 以重复模式（`--repeat-each=10`）运行检测不稳定
- **不稳定测试须在合并前修复，不允许提交跳过标记**

### CI/CD 集成
```
checkout → install → deps → test → upload artifacts
```
测试报告包含：摘要 + 失败详情 + 产物路径 + 健康评分

## 自调节机制

- 每轮修复后重跑受影响测试，验证修复效果
- 修复引入新失败：回退本次修复，重新分析
- **硬上限**：单次测试会话最多 15 次修复尝试，超过则停止并报告用户

## 跨会话学习

- 每次测试完成后，将发现的模式记录到项目 `.gstack/learnings.jsonl`（或项目约定的学习文件）
- 常见问题类型、高频失败页面、特定组件的易错点
- 后续测试会话加载学习记录，优先检查历史高频问题

- 键盘导航完整性
