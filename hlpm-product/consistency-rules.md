# 三项一致性检查规则与矩阵模板

> 属于 `hlpm-product` 技能第 9.5 步骤。
> 核心目的：确保 PRD / 设计稿 / 测试用例三项交付物逻辑完全一致。
> 开发段（`hlpm-dev`）以此为验证基准。

---

## 强同步规则（v5 核心）

**触发条件**：任一项因评审驳回而回退修改。

**强同步行为**：
1. 修改源项，版本号 +1
2. 检查另外两项是否需要同步修改
3. 同步修改受影响的项，版本号 +1
4. **强制重走后续所有评审**（不可豁免已通过的评审）
5. 评审通过后，重新生成一致性矩阵
6. 三项版本号重新对齐为新 vX

**触发示例**：

| 触发源 | 必须同步 | 必须重走评审 |
|-------|---------|------------|
| PRD 评审驳回 | 设计 + 用例 | 步骤 5/7/9（自 + 设计 + 用例） |
| 设计评审驳回 | PRD（如影响业务逻辑）+ 用例 | 步骤 5/7/9（如 PRD 改则加 5） |
| 用例评审驳回 | PRD（反查）+ 设计（反查） | 步骤 5/7/9（如 PRD/设计改则加） |

**3 轮回退上限**：单项评审最多驳回 3 轮，超出升级用户决策。

**兼容要求范围**：浏览器版本 / 设备类型 / 系统版本 / 屏幕尺寸

**填写规则（3 个子表通用）**：
- 每条非功能指标必须有：设计 UI 实现策略 + 测试验证方法
- **设计实现策略 = UI 层的体现方式**（如"骨架屏 / 虚拟滚动 / 错误脱敏 / 浏览器前缀"），**不写架构策略**（如"负载均衡 / 缓存层"——这些是技术设计文档范畴）
- 性能指标需含具体数字
- 安全需含具体策略（不能只写"安全"二字）
- 兼容性需含具体版本/设备列表
- 用例缺压测 / 安全 / 兼容性 = ❌

**简化场景（MVP 阶段）**：
- 性能：至少 1-2 条核心指标
- **安全：必含，绝对不可豁免**（密码 / 鉴权 / 注入防护 / CSRF / XSS）—— 即使 MVP 阶段也必须包含
- 兼容性：至少覆盖主用浏览器
- **豁免机制**（**仅性能 / 兼容性可豁免**）：
  - 如 MVP 阶段需豁免性能或兼容性某项，**由产品经理 `analyst` 决定**
  - 必须在不一致清单中记录豁免理由（如"业务方已确认 MVP 不做性能优化"）
  - **安全**不在豁免范围内，任何 MVP 阶段都不可豁免
  - 如遇特殊情况（如合规要求更高级别安全），升级到用户决策

### 5. 代码实现追踪矩阵(4 维防御基础）

**目的**：在产品段交付物基础上，**开发段用于追踪每条 PRD 规则的代码实现状态**。这是 4 维防御（漏实现 / 错误实现 / 非功能漏实现 / 范围越界）的统一追踪表。

**结构**（7 列，复用前 3 列 + 新增 3 列）：

| # | PRD 业务规则 / 状态机 / 权限 / 非功能 | 设计页面 | 测试用例 | 代码提交 SHA | 实现状态 | 偏离说明 |

| 3 | 优惠券不可叠加 | cart.html | TC-003 | ghi9012 | ⚠️ 偏离 | 实际允许叠加 1 张（错误实现 B 类） |
| 4 | 密码 bcrypt 加密（cost ≥ 10） | register.html | TC-S001 | jkl3456 | ⚠️ 偏离 | 实际用 MD5（非功能漏实现 C 类） |
| 5 | — | — | — | mno7890 | ➕ 越界 | 实现"商品收藏"功能但 PRD 未定义（范围越界 D 类） |
| 6 | ... | ... | ... | ... | ... | ... |

**实现状态枚举**（共 4 种 + 1 个 N/A）：

| 状态 | 含义 | 触发动作 |
|------|------|---------|
| ✅ 已实现 | 代码完全符合 PRD 规则 | 无 |
| ❌ 未实现 | PRD 有规则但代码未实现（场景 A） | 步骤 4.5 标记 + 步骤 7.5 走查不通过 |
| ⚠️ 偏离 | 代码实现了但语义不一致（场景 B/C） | 步骤 4.5 标记 + 步骤 7.5 走查 + 触发强同步 |
| ➕ 越界 | 代码实现了 PRD 未定义的功能（场景 D） | 通知产品段确认：保留则补 PRD / 移除则删代码 |
| N/A | 不适用（如仅设计相关） | 标灰跳过 |

**填写规则**：
- 代码提交 SHA 取最近一次修改该规则的提交（git log -p 按关键词搜）
- 偏离说明必填：必须含 PRD 条款引用 + 实际实现描述
- 每完成一个业务规则 / 状态机分支 / 非功能实现 → 立即更新（步骤 4.5）
- 走查阶段（步骤 7.5）逐行核对 → 全部 ✅ 才算走查通过

**与 v5 一致性矩阵的关系**：
- v5 矩阵（1-4）：产品段用，**PRD ↔ 设计 ↔ 用例**三方覆盖
- v10 矩阵（5）：开发段用，**PRD ↔ 代码**四方追踪
- 两表合起来 = "PRD 100% 实现保障"：v5 保证每条规则被设计/用例覆盖，v10 保证每条规则被代码实现

**MVP 豁免**：
- 小功能（< 3 个业务规则）可豁免第 5 矩阵，**仅走 v5 一致性矩阵**
- 豁免决定由产品经理 `analyst` 在 9.5 步骤做出，在不一致清单中记录豁免理由
- 大功能或核心模块**必须**填第 5 矩阵，无豁免

**与开发段拒收分类的对应关系**（P2-11 评审补充）：
- ✅ 已实现 ↔ handoff-rejection.md §9 无偏离项
- ❌ 未实现 ↔ **A 漏实现**（场景 A）
- ⚠️ 偏离（语义）↔ **B 错误实现**（场景 B）
- ⚠️ 偏离（非功能）↔ **C 非功能/安全漏实现**（场景 C）
- ➕ 越界 ↔ **D 范围越界**（场景 D）
- N/A ↔ 不适用，无需对应

## 提取方法（如何填充矩阵）

### 从 PRD 提取

1. 业务逻辑模块：每条 if-then 规则、每条边界判断、每条异常处理 = 1 条业务规则
2. 状态机模块：每个核心实体的所有状态 = 状态机行
3. 权限模块：每条"XX 角色不可/可 XX"规则 = 权限行

### 从设计稿提取

1. 每个 HTML 页面 = 1 个设计页面
2. 每页的 5 态（默认 / hover / 空 / 错 / 加载）= 状态机对应
3. 权限控制点：登录拦截、按钮可见性、页面访问限制 = 权限点

### 从测试用例提取

1. 每个 TC = 1 个测试用例
2. TC 描述中的"前置条件"应反推 PRD 业务规则
3. TC 的"预期结果"应与 PRD 验收条件一致

### 交叉比对

对每个矩阵：
- PRD 业务规则 N → 找设计页面 → 找测试用例 → 三者勾选一致 = ✅
- 任一缺失 = ❌
- 存在但内容不符 = ⚠️

## 一致性矩阵的演进

**每次重走评审后**：
- 修改源项 → 更新对应行
- 同步项 → 同步更新对应行
- 重新提取 → 更新版本号
- 重新走 9.5 步骤 → 输出新一致性矩阵

**版本一致性**：
- 一致性矩阵的版本号 = 当前三项版本号（vX）
- 强同步触发时，一致性矩阵版本号同步 +1

---

## 常见不一致模式（避免清单）

| 模式 | 表现 | 防范 |
|------|------|------|
| 业务规则遗漏 | PRD 写了规则但无测试用例 | 用例评审时强制反查 PRD |
| 状态机缺态 | PRD 定义 5 态但设计只画 3 态 | 设计评审时强制比对状态机 |
| 权限漏测 | 设计有权限控制但无反向测试 | 用例评审时强制要求正反向 |
| 字段不一致 | PRD 字段名 ≠ 设计字段名 ≠ 用例字段名 | 三方使用统一的字段术语表 |
| 交互流程偏差 | PRD 主流程 A → B → C，但设计稿显示 A → C | 设计评审时按 PRD 流程逐页核对 |
| 验收条件模糊 | PRD 验收"性能良好"，但无具体指标 | 非功能需求必须含具体 QPS / 响应时间 |

## 简化场景处理

**MVP 阶段**（v1 早期）：
- 业务规则矩阵可仅覆盖核心 5-10 条
- 状态机矩阵可仅覆盖主实体
- 权限矩阵可仅覆盖角色级（不细到字段级）

**成熟阶段**：
- 三个矩阵必须全量覆盖
- 每个矩阵行数 ≥ 业务复杂度对应数量
- 不一致项必须 100% 闭环

---

## 与开发段的衔接

开发段（`hlpm-dev`）步骤 0 验证 5 项交付物时，**一致性矩阵是核心验证项**：
- 一致性矩阵存在 + 全部 ✅ → 通过
- 一致性矩阵缺失 → 拒收（按"缺失"分类）
- 一致性矩阵有 ❌ → 拒收（按"三项不一致"分类）
- 三项版本号不匹配 → 拒收（按"三项不一致"分类）

## 自动化方向(建议中)

> 目标:将"人工填 SHA + 人工扫代码 + 人工对齐"流程**自动化**,降低人因失误,提升矩阵实时性
> 当前状态:本仓库已发布至更高版本,但以下 6 项自动化能力尚未落地,作为未来规划供有意向的团队/贡献者参考。

### 自动化-1. Git Hook 自动 SHA 回填(优先级 P0)

**触发时机**:推荐 `pre-push` 阶段(避免 `post-commit` 改文件触发循环 commit)

**实现**:

```bash
# 推荐: .git/hooks/pre-push  (推送前一次性写入,不入 commit 历史)
# 不推荐 post-commit: hook 改文件 -> 文件 dirty -> 阻断下次 commit -> 死循环
#!/bin/bash
# 扫描本分支所有 commit 修改的文件
CHANGED=$(git diff --name-only origin/main..HEAD)
# 对每个修改文件,扫描一致性矩阵的"对应文件路径"列
# 若匹配,自动填入 `git log -1 --format=%H -- <file>`
# 状态默认填 "⚠️ 待验证",需开发在 PR 中人工确认
# 写入工作区文件(不 commit),由开发者自行 commit
```

**前置条件**:
- 矩阵文件中"代码提交 SHA"列必须含"对应文件路径"列(当前未含,需扩展)
- 矩阵文件必须纳入版本控制(`docs/consistency-matrix.md` 或同等位置)

**收益**:
- 消除"开发忘记填 SHA"场景
- SHA 实时跟随 commit,无须人工维护

### 自动化-2. CI/CD 集成:PR 合并前自动跑覆盖率审计(优先级 P0)

**触发时机**:PR 提交 + target 分支更新时

**实现**:

```yaml
# .github/workflows/consistency-audit.yml
name: Consistency Matrix Audit
on: [pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0  # 需完整历史以算 SHA
      - name: Run Step 12.5 Audit
        run: |
          # 1. 提取一致性矩阵的所有"PRD 条款引用"
          # 2. grep 代码库,验证每条都有对应实现
          # 3. 输出覆盖率:已实现 / 应实现 = X%
          # 4. 覆盖率 < 95% → fail PR
```

**收益**:
- 拦截"漏实现"在 PR 阶段,无须等到生产审计
- 覆盖率指标可量化,杜绝"主观判断"

### 自动化-3. IDE 插件:实时显示矩阵状态(优先级 P1)

**目标 IDE**:VSCode(优先)、JetBrains 系列

**实现要点**:
- 侧边栏面板,显示当前文件所在模块的矩阵行
- 状态颜色编码:✅ 绿 / ❌ 红 / ⚠️ 黄 / ➕ 蓝
- 点击矩阵行 → 跳转到对应代码位置

**收益**:
- 开发在 IDE 中即可看到"我正在改的代码对应 PRD 哪条规则"
- 降低"实现完才发现偏离 PRD"的返工成本

### 自动化-4. 矩阵可视化(HTML 报告,优先级 P1)

**入口**:`make consistency-report` 或 `npm run consistency:report`

**输出**:`docs/consistency-report.html`,包含:
- 5 个矩阵的可视化表格(红黄绿状态着色)
- 覆盖率仪表盘(核心 100% / 一般 ≥95% / 工具 ≥90%)
- 偏离项清单(点击展开详情)
- 历史趋势图(最近 N 次审计的覆盖率变化)

**收益**:
- 评审会签时直接展示 HTML 报告,无须人工讲解
- 趋势图可识别"项目质量是上升还是下降"

### 自动化-5. 自动 Diff 检测(v5 遗留,优先级 P2)

**目标**:PRD 评审驳回时,自动定位"哪些设计/用例需要同步修改"

**实现**:
- 解析 PRD 变更 diff(git diff of `.hl/prd.md`)
- 关键词匹配 + LLM 辅助判断,定位受影响的:
  - 设计稿章节
  - 测试用例编号
- 输出"重走评审清单"给产品经理

**收益**:
- 自动化"强同步触发"判断,避免"凭感觉决定哪些要重走"

### 自动化-6. 跨项目设计规范合并(v4 遗留,优先级 P3)

**目标**:多项目共享同一套设计规范时,自动检测"本项目是否有自己的覆盖/偏离"

**实现**:
- 共享规范 → 公共仓库
- 本项目规范 → 项目仓库
- diff 检测 + 输出"偏离共享规范"清单

**收益**:
- 减少"每个项目重复维护一份设计规范"成本

### 实施优先级

| 优先级 | 功能 | 预计工时 | 依赖 |
|--------|------|---------|------|
| **P0** | Git Hook SHA 回填 | 1 人天 | 矩阵文件含文件路径列 |
| **P0** | CI/CD 覆盖率审计 | 2 人天 | 一致性矩阵格式稳定 |
| P1 | IDE 插件 | 5 人天 | 矩阵 API 化 |
| P1 | 矩阵可视化 | 2 人天 | 无 |
| P2 | 自动 Diff 检测 | 3 人天 | LLM API 可用 |
| P3 | 跨项目设计规范合并 | 5 人天 | 共享规范仓库 |

**实施前置(任一项满足即可启动对应自动化)**:
- 至少 1 次"人工漏填 SHA" / "人工漏跑覆盖率"导致的返工案例 → 启动自动化-1 或 自动化-2
- 团队规模 ≥ 3 人 → 启动 IDE 插件 / 可视化(自动化-3 / 自动化-4)
- 出现"凭感觉决定哪些要重走"的争议 → 启动自动化-5
- 多项目共享同一套设计规范 → 启动自动化-6
