# hlpm-product 技能说明(概览)

> **5 分钟读懂这个技能**
> 完整规范见 [`SKILL.md`](./SKILL.md),日常使用见 [`README.md`](./README.md),快速试用见 [`TRIAL.md`](./TRIAL.md)

## 一句话定义

**hlpm-product** 是 hlskills 体系中的"产品段",负责把需求变成**可交接的开发物**,让 `hlpm-dev` 直接接手写代码。

---

## 解决什么问题

### 没有这个技能时,产品+开发协作的痛点

```
产品经理                                    开发
   │                                        │
   │  "我想要个新功能,看这个 PRD"            │
   ├──────────────────────────────────────→ │
   │                                        │  "PRD 里业务规则不清楚"
   │  "我补充一下..."                       │  "字段定义是什么?"
   ├──────────────────────────────────────→ │  "状态机有几种状态?"
   │                                        │  "这个非功能指标怎么测?"
   │  "我再补充..."                          │
   │  ...几轮后...                           │
   │                                        │  "做完了,你看看"
   │  "这跟我想的不一样"                     │
   │  ...再来几轮...                         │
```

**典型问题**:
- 产品 PRD 写得不完整,开发来回问
- 设计稿和 PRD 对不上,实现完才发现
- 测试用例和需求脱节,验收时扯皮
- 一方改了文档,另一方不知道,版本不一致
- 出问题不知道谁负责

### 用了这个技能后

```
产品经理 (hlpm-product)                开发 (hlpm-dev)
   │                                   │
   │  启动产品段                        │
   │  3 步生成 8 类文档                 │
   │  走完 3 场会签                     │
   │  自检 + 一致性矩阵                 │
   │  打包交付 ↓                        │
   ├──────────────────────→             │
   │  docs/v1/                          │
   │  ├─ prd.md                         │  启动开发段
   │  ├─ test-cases.md                  │  步骤 0 验证交付物
   │  ├─ design/                        │  通过 → 接手写代码
   │  ├─ acceptance-criteria.md         │  不通过 → 拒收 + 原因
   │  ├─ consistency-matrix.md          │  → 产品段补齐
   │  └─ ... (8 项)                     │
   │                                   │
```

**核心改变**:
- ✅ 一次到位,产品段输出**完整可验收**的 8 类文档
- ✅ 一致性矩阵自动校验三方对齐(PRD ↔ 设计 ↔ 用例)
- ✅ 评审会签强同步:任一文档改了,其他文档强制同步
- ✅ 拒收机制:开发可拒收不合格交付物,带拒收原因
- ✅ 版本目录管理:历史可追溯,新旧需求不冲突

## 核心机制(4 大支柱)

### 1. 角色边界铁律(防止越界)

```
产品段 (hlpm-product)              开发段 (hlpm-dev)
─────────────────                 ─────────────────
只产出文档                          只改代码
PRD / 用例 / 设计稿 / 验收          src/ / 配置文件 / 部署
不改任何源代码                     不产出业务文档
```

**铁律**:产品段绝不修改 `.ts/.js/.py/.java/...` 等源代码。改代码是 `hlpm-dev` 的事。

### 2. 23 步主流程(可裁剪)

完整 23 步覆盖 5 个阶段:
- **第 0 阶段**:加载上下文 + 规模判断 + 版本选择
- **第 1 阶段**:需求 → 竞品 → PRD
- **第 2 阶段**:设计(条件)
- **第 3 阶段**:用例
- **第 4 阶段**:交付前终检 + 交付

**裁剪规则**(按需跳过):
| 条件 | 跳过 |
|------|------|
| 轻量需求 | 竞品分析 + 完整第二阶段 + 部分矩阵 |
| 不涉及设计 | 整个第二阶段 |
| 选 3 合 1 评审 | 中间 3 场评审(合并为 1 场) |

**最快路径**:12 步(轻量 + 不涉及设计 + 3 合 1)

### 3. 评审会签(防返工)

**2 种模式**(用户启动时选):
- **联合评审**(重量):3 场独立会签,适合复杂需求
- **3 合 1**(轻量):1 场集中评审,适合小需求

**强同步规则**:**任一文档因评审驳回而修改,后续所有评审必须重走**(不能豁免已通过的)

### 4. 一致性矩阵(防三方对不上)

```
PRD 业务规则 ──┐
               ├──→  业务规则覆盖矩阵  ──→ 不一致清单
设计稿页面 ────┤        (✅/❌)
               │
测试用例 TC ───┘
```

**5 个矩阵**:业务规则 / 状态机 / 权限 / 非功能 / 代码实现追踪
**轻量模式只跑前 2 个**

| 技能 | 关系 | 何时用 |
|------|------|--------|
| **`hlpm-dev`** | 配套(强) | 产品段交付后,开发段接手 |

| `hlreview` | 后置(弱) | 详细评审规范参考 |

**核心配套**:`hlpm-product` + `hlpm-dev` = 完整产品+开发协作流程

## 适用 vs 不适用

### ✅ 适用场景

- **多人协作**:团队有产品+开发两个角色
- **质量门**:需要"评审会签"防返工
- **可追溯**:需要历史版本管理
- **强一致**:要求 PRD/设计/用例三方完全对齐
- **可拒收**:允许开发对不合格交付物退回

### ❌ 不适用

- **单人小团队**:只有你一个人,产品+开发都是你 → 用 `hlpm` / `hlpmnew`
- **无评审机制**:团队不想要会签流程 → 用 `hlpm` 更轻
- **临时小修复**:1 行 bug 修复 → 直接改代码即可,不需要产品段
- **新项目 0→1**:要看具体情况,可能 `hlpmnew` 更合适

## 3 个关键设计决策(为什么这样做)

### 决策 1:产品段**绝对不写代码**

**理由**:
- 角色清晰:产品=业务,开发=实现
- 可追溯:文档修改可 review,代码修改可 blame
- 可拒收:开发可对"代码不符合 PRD"明确说"不"

**代价**:小需求也要走流程(避免直接动手)

### 决策 2:**版本目录**而不是平铺

**理由**:
- 历史可查:每次产品段流程独立目录
- 不会覆盖:v2 不会破坏 v1
- 拒收友好:开发拒收 v2,产品段在 v2 上补齐

**代价**:多一层目录(`docs/v1/` vs `docs/`)

### 决策 3:**强同步规则**而非"评审通过即可"

**理由**:
- 防止"局部改动"破坏整体一致
- 例:用例改了 → PRD 验收标准可能要改 → 设计交互可能要改

**代价**:一次驳回要重走多个评审,慢

## 关键数字

| 指标 | 值 |
|------|---|
| 完整步骤数 | 23 步 |
| 最少步骤数 | 12 步(轻量 + 不涉及设计 + 3 合 1) |
| 输出文档数 | 7-8 项(条件决定) |
| 评审会签最多 | 3 场(联合评审) / 1 场(3 合 1) |
| 一致性矩阵 | 5 个(完整) / 2 个(轻量) |
| 单项评审驳回上限 | 3 轮 |
| 3 合 1 不通过自动升级 | 是(转联合评审) |
