## 版本号管理（v5 强制）

**PRD / 设计稿 / 测试用例 三项必须版本号一致**才能交付：

```
当前交付版本：vX（X ≥ 1）
- PRD vX
- 设计 vX
- 用例 vX
```

版本变更触发条件：
- 任一项评审驳回导致内容修改 → 该项版本号 +1
- 触发【强同步】→ 其他两项必须同步 +1，三项版本号重新对齐
- 一致性矩阵每次重新生成时记录当前版本号

## 自检不合格的处理

| 自检发现 | 处理方式 |
|---------|---------|
| PRD 模糊字眼 | 返回步骤 4 补齐 + 重走 PRD 评审 |
| 设计稿违反规范 | 返回步骤 6b 补齐 + 重走设计评审 |
| 用例与 PRD 不一致 | 返回步骤 8 补齐 + 重走用例评审 + 触发强同步 |
| 三项版本号不一致 | 按一致性矩阵核对，回退到落后项 + 触发强同步 |
| 验收标准与 PRD 冲突 | 返回步骤 10 重新打包 |
| 一致性矩阵 ❌ 项未处理 | 返回对应步骤（4 / 6b / 8）修改 + 触发强同步 |
| 非功能需求缺指标 | 返回步骤 10 补齐具体数字 |

## 强同步触发后的版本管理

强同步触发时，按以下顺序更新：

1. 确认触发源（如：PRD 评审驳回）
2. 修改源项，版本号 +1
3. 同步修改受影响的另外两项，版本号 +1
4. 重走后续所有评审
5. 评审通过后，重新生成一致性矩阵
6. 三项版本号重新对齐为新 vX

**禁止**：仅更新源项版本号而不重走其他评审。
**禁止**：版本号跳号（如 v1 直接变 v3）。
