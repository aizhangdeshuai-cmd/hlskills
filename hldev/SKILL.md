# 多角色协作-开发段流程

> ## ⚠️ 流程纪律的执行机制
>
> 本流程的"硬性关卡""返回上一步""强同步规则"是 markdown 文本纪律,Claude Code 不会机械地强制执行。Agent 加载后会**自觉遵守**,但如 Agent 模型弱或 prompt 冲突,关卡可能失效。
>
> **真实约束**:文档本身 + Agent 自觉 + 用户手动打断。
> **不是**:Claude Code runtime hook 拦截 + 状态机门禁。
>
> 如发现 Agent 在第 X 步没停下来等你确认 / 直接跳过第 Y 步 / 没返回上一步重做,请**手动打断**让它重读 SKILL.md 对应章节。
>
> 详细能力边界见仓库 `README.md` 「⚠️ 能力边界声明」段。


> 属于 `hlskills` 技能系统。与 `hlpm` 配套使用，构成"产品 + 开发"分工的标准开发流水线。
> 完整规范见 `hlskills/SKILL.md`。

## 通用纪律：完成前验证

**任何阶段标记"完成"前，必须执行验证规则：**
- 识别能证明该阶段完成的验证命令，完整运行
- 读取完整输出并检查退出码，确认通过后方可声明完成
- **绝对禁止**："应该通过了"、"看起来正确"、"可能没问题"
- **铁律**：证据先于断言，无一例外

### 受阻停止纪律
- 任何步骤遇到阻塞，**立即停止**，不得猜测或强行绕过
- 向 `executor` 报告阻塞
- 3 次修复尝试失败 = 质疑架构方案，升级到用户决策
- 交付物拒收 = 退回 `hlpm`，不自行猜测

### 拒收纪律

**开发段有权拒收不合格的交付物，且必须给拒收原因**：
- 不符合 → 填写 `handoff-rejection.md`（具体条款 + 原因）→ 退回产品
- 拒收不是"失败"，是"质量门"
- 退回产品段后，产品段需按强同步规则补齐 + 重走评审

## 角色与 Agent 映射

| 角色 | Agent | 用途 |
|------|-------|------|
| 前端开发 | `executor` | 交付物验证、前端实现 |
| 后端开发 | `architect` / `executor` | 后端架构设计与实现、ADR 编写 |
| 代码审查 | `code-reviewer` | 代码质量审查（严重 / 重要 / 次要三级） |
| 安全审查 | `security-reviewer` | 安全漏洞扫描 |
| 测试 | `test-engineer` / `qa-tester` | 用例执行、浏览器验证 |
| 验证 | `verifier` | 生产审计、交付验证 |
| 用户（我） | — | 发布 / 部署确认者（硬性关卡） |

## 开发段开发流程（15 步）

### 第零阶段：交付物验证

0. **★ 验证版本目录 + 8 项交付物 + 一致性矩阵**（`executor` 主导，阻塞点）
   - 接收来自 `hlpm` 步骤 12 的交付包
   - **:验证版本目录**
     - 确认交付物位于 `docs/vN/` 目录(而非散落在 `docs/` 根)
     - 扫描项目根目录:`ls -d docs/v*/ 2>/dev/null | sort -V`
     - 找到含 8 项交付物(PRD/测试用例/验收/非功能/一致性/自检/设计)的最新版本目录
     - **vN 标记文件检查**:
       - `docs/vN/.dev-completed` 存在 → 已被接手/发布,**当前验收的是更新的版本**
       - `docs/vN/.dev-rejected` 存在 → 已被开发段拒收,**当前验收的是补齐版本**
       - `docs/vN/.product-archived` 存在 → 已归档,**跳过该版本**
   - 验证清单（来自 `handoff-package.md`）：

   ```markdown
   ## 版本目录 + 8 项交付物验证

   ### 版本目录（v12 必查）
   - [ ] 交付物位于 `docs/vN/` 目录(N 为版本号)
   - [ ] 8 项交付物齐全:prd / test-cases / acceptance-criteria / non-functional-requirements / consistency-matrix / handoff-self-check / design/*.html / analysis/competitive-analysis.md
   - [ ] 标记文件状态:无 .dev-completed (未被接手) / 无 .dev-rejected (未被拒收)

   ### 版本一致性（必查）
   - [ ] PRD 版本：vX
   - [ ] 设计版本：vX
   - [ ] 用例版本：vX
   - [ ] 三项版本号匹配（vX = vX = vX）

   ### 一致性矩阵（必查）
   - [ ] 一致性矩阵存在
   - [ ] 业务规则覆盖矩阵完整
   - [ ] 状态机覆盖矩阵完整
   - [ ] 权限覆盖矩阵完整
   - [ ] 非功能需求覆盖矩阵（性能 / 安全 / 兼容 3 个子表）完整
   - [ ] 不一致清单为空或已闭环
   - [ ] 全部 ✅ 通过

   ### 自检报告（v6 必查）
   - [ ] 自检报告存在（`docs/{ver}/handoff-self-check.md`）
   - [ ] 自检报告与 handoff-package.md 自检清单勾选一致
   - [ ] 自检报告签字方完整

   ### 8 项产品交付物自检
   - [ ] PRD 章节结构合规(按"功能点剖切"组织:§0 上下文 + §1-§N BL-N 卡片 + 附录 A-F, 详见 hlpm SKILL.md §4 PRD 编写)
   - [ ] PRD 每张 BL-N 卡片含 14 字段模板(用例编号/名称/优先级/角色/前置/触发/主功能/拓展/业务逻辑/关键决策/埋点/验证/频率/NFR/未决)
   - [ ] PRD 6 大模块齐全(业务逻辑/操作流程/数据流转/状态机/权限/非功能) + 数据埋点 + 原型截图引用
   - [ ] PRD 无模糊字眼 + 字段级定义 + 评审会签
   - [ ] 设计稿符合项目设计规范 + 保真度档位 + 评审会签
   - [ ] 测试用例覆盖三维（业务规则 / 状态机 / 权限）+ 评审会签
   - [ ] 验收标准每条可验证
   - [ ] 非功能需求含具体指标
   ```

   - **🚨 设计稿搬移白名单(硬性)** — 设计稿 HTML 中含有"设计元信息"(用于评审),搬到生产代码时**必须剔除**。白名单:
     - **DOM 排除**:`<!-- DEV-NOT-FOR-PROD -->` 注释包裹的所有内容 / `.logic-*` 类 / `.floating-meta-*` 类 / `#floatingMetaToggle` / `#floatingMetaPanel` / `#logicPanel` / `#logic-data` 整个 `<script>`
     - **属性排除**:元素上 `[data-prd]` / `[data-tc]` / `[data-state]` / `[data-matrix]` / `[data-diff-mode]` / `.logic-badge` 类(逻辑说明角标)
     - **CSS 排除**:`.logic-*` / `.floating-meta-*` / `.diff-add` / `.diff-mod` / `.diff-del` 选择器(评审视觉辅助)
     - **保留**:页面主体 DOM(区块 3 + 区块 3.1)+ 实际组件样式
   - **🚨 设计稿 5 区块扩展识别** — 如设计稿含 5 区块扩展(区块 3.2-3.5 弹窗在悬浮框 tab 内),**仅保留区块 3 + 3.1 DOM** 作为生产代码,其余 4 个区块禁止搬到生产。
   - **白名单来源**:与 `hlpm` SKILL.md 第 6b 步"5 区块扩展 DEV-NOT-FOR-PROD"段对齐。任何开发与产品对搬移范围有分歧 → 优先以本段为准。

   - **全部 ✅ → 进入步骤 1**
   - **任一 ❌ → 填写 `handoff-rejection.md`（拒收原因 + 不符合项）→ 退回产品段**
     - ****:在 `docs/vN/` 创建空文件 `touch docs/vN/.dev-rejected`（标记该版本已被拒收,产品段补齐后再次启动会知道该版本已拒收需重新走 9.5 终检）
   - 拒收原因分类（v5）：
     - 缺失（产品漏交某项）
     - 不达标（PRD 模糊字眼 / 设计违反规范 / 用例覆盖率低 / 非功能需求缺指标）
     - 内部矛盾（文档间冲突）
     - **三项不一致**(版本号不匹配 / 一致性矩阵有 ❌）

### 第一阶段：Git 与开发准备

1. **Git 工作区准备**
   - 检测当前环境（普通仓库 / 工作区）
   - 如在主分支，先创建 feature 工作区分支
   - 安装依赖并验证干净基线（运行现有测试确保通过）

2. **开发计划**（`planner`）
   - 制定可执行工作计划
   - 任务拆分粒度：单一步骤（2-5 分钟可完成）
   - 每个任务含：精确文件路径、完整代码、具体命令
   - 任务执行遵循受阻停止纪律

3. **架构决策记录**（`architect`）
   - 对关键技术选型以 ADR 格式记录到 `docs/adr/`
   - 接口设计遵循 `hlapi` 规范
   - 数据库迁移遵循 `hldb` 规范
   - 错误处理遵循 `hlerror` 规范

### 第二阶段：开发与审查

4. **开发**（`executor`）
   - 遵循技术设计文档、编码标准（`hlcode`）、API 设计规范、数据库迁移规范、错误处理规范、无障碍规范
   - 推荐 TDD，覆盖率 ≥ 80%
   - **并行策略**：前端和后端任务如无依赖关系，可并行分派
   - 多个独立模块可同时分派给多个 `executor` 实例
   - 严格遵循产品段的设计稿（保真度 + 设计规范）

4.5. **★ PRD 逐项勾选**(`executor` 主导，拦截漏实现 A 类）
   - **触发时机**：开发**过程中**，每完成一个业务规则/状态机分支/非功能实现并 **commit** 后 → 立即更新第 5 矩阵的 SHA 列（代码未 commit 则 SHA 不存在，无法填写）
   - **PRD 引用方式(按 hlpm v16 章节结构)**: PRD 按"功能点剖切", 每条业务规则对应 `§N` BL-N 卡片(详见 hlpm SKILL.md §4 PRD 编写)。一致性矩阵第 5 矩阵的"PRD 条款引用"列填 `§N.9` 或 `§N.12` 等卡片内子节编号, 而非"§1.BL-12"这种旧式维度章节编号
   - **操作**：更新一致性矩阵第 5 矩阵（`docs/consistency-matrix.md` §5，又名「代码实现追踪矩阵」）
     - 填写：代码提交 SHA（`git log -1 --format=%H -- <file>` 取最近一次修改；如一规则跨多 commit，取最后一个相关 commit 的 SHA）+ 实现状态（✅/❌/⚠️/➕）
     - 偏离说明必填：含 PRD 条款引用 + 实际实现描述
   - **越界检测**：如发现代码实现 PRD 未定义的功能 → 立即通知产品段确认
   - **MVP 豁免**：豁免决定已在产品段 9.5 步骤做出（见 `docs/consistency-matrix.md` §6 不一致清单），开发段只需查询结论；小功能（< 3 个业务规则）豁免后无需逐项勾选
   - **目的**：把"实现后检查"前移到"动手中勾选"，避免技术债累积
   - **与 7.5 走查关系**：4.5 边做边勾，7.5 全部完成后逐行核对（4.5 是录入，7.5 是审计）
   - **产出物路径**：`docs/consistency-matrix.md`（第 5 矩阵）+ 必要时 `docs/code-checklist.md`（勾选过程留痕）

5. **代码审查**（`code-reviewer`）
   - 审查代码质量，按严重 / 重要 / 次要三级输出
   - 严重问题：必须修复后才能合入
   - 重要问题：建议修复
   - 次要问题：可选优化

6. **前后端联调**
   - 前后端对接接口
   - 联合调试，确保接口对齐

7. **前后端自测**（`executor`）
   - 开发自测功能完整性与接口正确性
   - 按测试用例自测核心路径

7.5. **★ PRD 走查**(`executor` + `code-reviewer` 联合，拦截错误实现 B + 范围越界 D）
   - **触发时机**：开发**完成后**，分支完成前
   - **操作**：对照 `docs/consistency-matrix.md` 第 5 矩阵（代码实现追踪矩阵），**逐行核对代码实现**
     - 每行确认：实现状态（✅/❌/⚠️/➕） + 偏离说明（如有）
     - 输出**走查报告**：每行结果 + 偏离原因 + 修复计划
   - **走查不通过**：
     - ❌ 未实现 → 返回步骤 4 补齐 + **清空旧 SHA + 状态改为 ⚠️(待重审)**（防旧数据残留误判）
     - ⚠️ 偏离 → 返回步骤 4 重写 + **清空旧 SHA + 改为 ⚠️** + 触发强同步（v5 规则：不可豁免已通过的评审）
     - ➕ 越界 → 与产品段确认：保留则补 PRD / 移除则删代码
   - **走查通过**：进入步骤 8 分支完成
   - **MVP 豁免**：小功能可豁免，**仅走测试用例自测（步骤 7）**

### 第三阶段：分支收尾

8. **分支完成**
   - 代码审查通过后执行分支收尾
   - 检测环境 → 验证测试通过 → 确定基础分支
   - 呈现 4 选项给用户：
     - 本地合并
     - 推 PR
     - 保持原样
     - 丢弃
   - 执行用户选择 → 清理工作区

### 第四阶段：测试与审计

9. **安全审查**（`security-reviewer`）
   - 安全漏洞扫描
   - 输出风险等级与修复建议

10. **测试用例执行**（`qa-tester`）
    - 按测试用例执行（单元 + 集成 + E2E）
    - 验证产品段交付的验收标准全部通过

11. **浏览器验证**（`qa-tester`）
    - 浏览器环境验证
    - 无障碍测试（WCAG 2.2 AA）
    - 移动 / 平板 / 桌面三屏验证

12. **生产审计**（`verifier`）
    - 结构化审计
    - 输出风险评分（0-100）
    - 评分 < 50 → **受阻不可发布**

12.5. **★ PRD 覆盖率审计**(`verifier` 主导，拦截非功能漏实现 C + 兜底 A/B/D）
   - **触发时机**：生产审计**前**，上线前最后一道关卡
   - **操作**（人工 + 矩阵双轨）：
     - **主方式**：对照 `docs/consistency-matrix.md` 第 5 矩阵（代码实现追踪矩阵）逐行核对实现状态，已实现(✅)行数 / 应实现行数 = 覆盖率
     - **辅助线索（可选）**：可用 Grep 搜索关键词（如状态名 / 业务规则名）定位候选实现位置；正则/搜索结果**仅作线索，不作"已实现"判定依据**
     - **不依赖**：纯正则匹配控制流关键字（`if` / `switch` / `case`）作为"已实现业务规则数"的统计——这种推断不可信，因注释/字符串/任意控制流都会匹配上
   - **覆盖率要求**：
     - 核心模块（订单 / 支付 / 安全）：**100%** = "每条业务规则必须经 `verifier` 在一致性矩阵第 5 矩阵标注 ✅ + **人工复核签字**(产品经理或安全工程师)"。单一自动化检查不构成 100% 通过——LLM 没法机械判定"某条规则算不算核心模块",**最终判定依赖人工签字**
     - 一般功能:**verifier 标注 ✅ 即可** + ≥ 95% 阈值(作为参考,不作硬阻断)
     - 工具类 / 边缘功能:**verifier 标注 ✅ 即可** + ≥ 90% 阈值
   - **覆盖率不达标**:
     - 输出**差异清单**(每条:PRD 条款 / 实际状态 / 修复建议)
     - 强制返回步骤 4 补齐,**不能发布**
     - **3 轮回退上限**:连续 3 次不达标必须升级到用户决策(避免与 7.5 / 12.5 死循环)
   - **生产审计评分**(人工签字后):
     - 核心模块 100% 签字 → 加 5 分
     - 一般功能 ≥ 95% → 不加分
     - 一般功能 < 95% 或工具 < 90% → 每低 5% 扣 10 分
   - **MVP 豁免**:小功能可豁免,**仅走生产审计(步骤 12)**

12.7. **🚨 项目知识库完整性自检**(`executor` 主导, **阻塞发布**)
   - **触发时机**: 步骤 12.5 完成后, 步骤 13 发布前
   - **检查清单**(任一缺失 → 阻塞发布, 必须补齐):
     - [ ] `.hl/knowledge/api/{module}.md` 覆盖本版本所有新增/修改/删除接口(URL / Method / 入参 / 出参 / 错误码 / 权限)
     - [ ] `.hl/knowledge/db/{table}.md` 覆盖本版本所有新增/修改/删除表 + 字段(与 `migrations/` SQL 一致)
     - [ ] `.hl/knowledge/db/er-diagram.md` 反映最新表关系(如有外键变化)
     - [ ] `.hl/knowledge/adr/{NNNN}-{slug}.md` 含本版本所有重大决策(如本版本新增了 1 个 BL-N 3 方案选 1 个, 必有 ADR)
     - [ ] `.hl/knowledge/state-machines/{entity}.md` 反映最新状态机(本版本新增/删除状态/转移)
     - [ ] `.hl/knowledge/enums/{enum}.md` 覆盖本版本所有新增/修改枚举值
     - [ ] `.hl/knowledge/error-codes/README.md` 含本版本所有新增/修改错误码
     - [ ] `.hl/knowledge/dependencies/README.md` 与 `pom.xml` / `package.json` 一致
     - [ ] `.hl/knowledge/env-vars/README.md` 含本版本所有新增/修改环境变量
     - [ ] `docs/master-prd.md` 含本版本章节(如 v3 章节)
     - [ ] `docs/master-test-cases.md` 含本版本章节
     - [ ] 项目根 `CLAUDE.md` 反映最新项目元信息(技术栈 / 架构 / 当前版本)
   - **同 commit 规则**: 本次发布的代码 / 知识库条目 / master 文档必须**同次 commit 提交**。不允许"代码已发布, 知识库下次再说"
   - **失败处理**: 任一缺失 → 阻塞步骤 13 发布 + 步骤 4.5 重新走
   - **详细规则**: 详见 `hlkb` 技能 SKILL.md §同步触发表

### 第五阶段：交付与发布

13. **交付验证**（`verifier` 主导：创建完成标记）
    - 验收所有文档与功能完整性
    - 验证 14 项项目文档齐全（按 hlskills 主入口清单）
    - **:创建版本完成标记**:
      - 验收通过后,在 `docs/vN/` 创建空文件 `touch docs/vN/.dev-completed`
      - **严禁**验收不通过时创建(避免误标)
      - **严禁**删除已存在的 .dev-completed(让历史版本保持已开发状态)
    - 创建后通知:该版本已标记为"已开发",后续产品段启动会跳过此版本

14. **用户确认发布**（🚨 硬性关卡）
    - 向用户展示：版本号 / 变更日志 / 预发布清单
    - **用户未确认 → 不得推送标签**
    - 严禁自动发布

15. **用户确认部署**（🚨 硬性关卡）
    - 向用户展示：部署策略 / 目标环境 / 版本号 / 回滚方案
    - **用户未确认 → 不得部署**
    - 严禁自动部署

## 交接段

>
> 如果用户是**单独调用本技能**, 此交接段不触发, 由用户决定下一步。
