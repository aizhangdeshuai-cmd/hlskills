# Bug修复流程

## 通用纪律：完成前验证
> **路径规范**：本文件涉及的 `docs/` 路径命名遵循 `hlpm-product/path-conventions.md` 中央规范（命名/深度/特例）。


**任何阶段标记"完成"前，必须识别并运行验证命令，读取输出并确认退出码通过。** 严禁"应该通过了"、"看起来正确"、"之前检查过了"、"可能没问题"等说法。证据先于断言，无一例外。未经新鲜验证证据，不得声称完成。

### 受阻停止纪律
- 任何步骤遇到阻塞，**立即停止**，不得猜测或强行绕过
- 3 次修复尝试失败 = 质疑架构方案，不尝试第 4 次变体修复

### 自调节机制
防止无限修复循环：
- 每轮回退 +15% WTF 概率
- 单次修复涉及 > 3 个文件 +5%
- 无关文件变更 +20%
- **硬上限：15 次修复后停止**，汇总问题向用户报告
- 修复引入新问题：回退本次修复，标记为"需人工介入"

### 调试纪律（融合 gstack /investigate）

**根因未确认前，禁止进入修复阶段。** 具体规则：
- 必须先完整收集证据（错误日志、堆栈、复现条件、环境信息），才能提出假设
- 每个假设必须可验证——"可能是 X 导致"不算验证，必须有具体验证步骤
- **症状 ≠ 根因**：console 报错是症状，为什么报错才是根因。修复症状不修复根因 = 埋下未来 Bug
- 对复杂 Bug 使用 `tracer` Agent 做因果追踪，对竞态条件、异步问题的排查优先级高于直觉判断
- 跨组件 Bug 先画调用链，再逐层排查

## Bug修复流程

### 第零阶段：加载项目上下文
0. **加载已有文档** — 修复之前先理解项目，避免孤立分析：
   - 如存在 `.hl/memory/` → Read 全部文件
   - 如存在技术设计文档 / API文档 / 数据库设计 → Read 加载相关模块
   - 如 Bug 涉及已分析过的模块 → 加载对应分析文档

### 第一阶段：问题定位
1. **Bug分析** — `debugger` 基于项目文档上下文理解Bug现象与影响范围。如用户提供Bug截图/录屏：优先用模型原生多模态识别，不支持则用 Read 工具读取图片；如需分析视频可使用可选外部 `videodb` 技能（需用户自行安装，未安装不影响本流程）
2. **根因定位** — `architect` + `executor` 定位问题根因。遵循结构化调试四阶段（融合 gstack /investigate 方法论）：
   - **Investigation（证据收集）**：收集所有相关证据——错误日志、堆栈跟踪、复现条件、受影响范围
   - **Analysis（根因分析）**：从证据出发，追踪因果链路，定位根因而非症状
   - **Hypotheses（假设验证）**：对每个候选根因提出可验证假设，逐个排除
   - **铁律：根因未确认前，禁止进入修复阶段**。修复症状而不修复根因 = 隐藏未来 Bug
3. **修复方案** — `architect` + `executor` 制定修复方案
4. **用户确认方案** — 提交修复方案供用户确认

### 第二阶段：修复与验证
5. **Git 工作区准备** — 创建隔离工作区进行修复，避免污染主工作区
6. **修复开发** — `executor` 执行修复，遵循技术设计文档与编码标准
7. **修复自测** — 开发自测修复正确性，确保无回归
8. **代码审查** — `code-reviewer` 审查修复代码，确保无引入新问题
9. **安全审查** — `security-reviewer` 审查修复是否引入安全风险
10. **回归测试** — `qa-tester` 对受影响模块执行回归测试
11. **浏览器验证** — `qa-tester` 浏览器环境确认Bug已修复
12. **分支完成** — 执行分支收尾：检测环境→验证测试→确定基础分支→4选项（本地合并/推送PR/保持原样/丢弃）→执行→清理
13. **最终验证** — `verifier` 验证修复完整性与安全性
14. **交付** — 更新相关文档，记录修复内容

## 变更传播规则

Bug修复涉及逻辑变更时，**受影响文档必须同步更新**：接口文档、数据库设计、测试用例、用户操作手册、帮助文档。确保代码逻辑与文档描述保持一致，不得出现文档与实现不符的情况。
